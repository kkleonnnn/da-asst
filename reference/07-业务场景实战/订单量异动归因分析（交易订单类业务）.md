## 正文

### 一、一句话方法论

订单是这类业务的主脉络，绝大多数问题最终都会落到「订单为什么这么多/这么少」。回答它不靠上复杂模型，而是一条朴素的「长公式」：**先用时间序列处理消掉周期与噪声、建立「任意时间点订单应该是多少」的基准 → 再用同比/环比把起伏曲线浓缩成几个百分点的真实差距 → 最后沿「订单类型 × 地理区域」做链式拆解、配两段式钻取，把波动归因到具体人群与具体城市。** 拆解逻辑负责「往下切」，钻取逻辑负责「盯住一类指标看深」，两者合起来回答「问题到底出在哪、存在多久」，给出可信度约八成的结论，剩下靠业务一起确认。

### 二、第一步：时间序列预处理，目的是「还原业务」不是套公式

拉出全年订单时序图，通常是一条带周期波动、整体先升后降（或先降后升）的锯齿曲线。直接拿它做对比或拟合会被周期和异常带偏，先处理三类问题。

**1. 极端值与缺失值**

- **删除**：节假日、极端天气会让订单骤降甚至跌到 0，这类异常点直接参与同环比或建模会污染结果，正常可直接删掉。
  ```python
  df.dropna(subset=['value'], inplace=True)
  ```
- **填充 / 插值**：删多了会丢信息、破坏序列完整性。填充的关键不是机械取均值，而是**用「本应相似」的业务信息去补**。例如删掉某个异常的周一，应优先参考上一周的周一、或近期天气相近的日子来填，而不是拿相邻的周日、周二做移动平均。
  ```python
  df['value'].fillna(df['value'].mean(), inplace=True)               # 均值填充（会引入偏差）
  df['value'].interpolate(method='linear', inplace=True)             # 线性插值：简单，抓不住真实变化
  df['value'].interpolate(method='polynomial', order=2, inplace=True)# 多项式：能抓局部变化，但易过拟合
  df['value'].interpolate(method='pchip', inplace=True)              # 单调分段三次插值：更准，但段交接处可能不连续
  ```
  注意：若填充后的数据还要拿去建模预测，本质是「用预测结果再预测」，且被处理过的数据已不再反映真实业务，解释结论时要记住这一点。

**2. 周期性**

用户行为天然有周期（如工作日多、周末少），曲线呈锯齿状。处理方式：
- **差分**：一阶差分（当前值减前一个观测值）消除线性趋势；一阶不够再二阶。企业内一般用不到二阶，除非数据常年线性增长。
  ```python
  df['value_diff'] = df['value'].diff()
  ```
- **周期/季节调整**：数据减去对应周期的均值（一周均值、工作日均值、季节均值等），思路与差分同源，更常用。
  ```python
  df['value_seasonally_adjusted'] = df['value'] - df['value'].rolling(window=7).mean()  # 假设周期 7 天
  ```
- **季节性模型 SARIMA**：可同时考虑趋势、季节性和噪声。
  ```python
  from statsmodels.tsa.statespace.sarimax import SARIMAX
  model = SARIMAX(df['value'], order=(1, 1, 1), seasonal_order=(1, 1, 1, 7))
  model_fit = model.fit()
  ```

**3. 数据转换（应对分布偏斜 / 不满足统计假设）**

| 转换 | 适用 | 要求 |
|---|---|---|
| 对数转换 | 右偏数据，压缩尺度、减小偏斜，使分布更接近正态 | 数据必须为正 |
| 平方根转换 | 右偏数据，效果通常比对数温和 | 数据不能为负 |
| 反转换（倒数） | 左偏数据，扩大尺度、减小偏斜 | 注意取值范围 |

```python
df['value_log']  = np.log(df['value'])
df['value_sqrt'] = np.sqrt(df['value'])
df['value_inv']  = 1 / df['value']
```
转换后所有分析都基于变换后的尺度，解释时要换算回来；被转换的数据已不再直接对应真实业务量。

> 关于移动平均、指数平滑、AR、MA、ARMA、ARIMA 等经典时序方法：作为通识了解即可。实际企业数据复杂、不满足分布假定，工作中大多用不上；要紧的是理解其本质（用历史值/历史误差的加权刻画惯性），从而在真实工作里形成自己的处理策略。

### 三、第二步：用同比/环比建立「应该是多少」的基准

时序图再平滑，也只解决了「曲线长这样」；真正回答「这个点的订单应该是多少」靠的是对比。

- **同比（YoY）**：与去年同期比，能消掉大部分季节性影响（重点核对春节等节点是否对齐，否则同比也会失真）。
- **环比（本期比上期）**：与上一期比，看短期连续变化。

把去年曲线「印」在今年曲线下方，起起伏伏的差异会浓缩成个位数的百分点——**那几个百分点才是今年的新问题**。一句话：如果什么都不变，今年的曲线应该和去年一样。由此延伸三个常用指标，各回答一个问题：

| 指标 | 回答的问题 |
|---|---|
| 年同比（今年 vs 去年） | 今年订单和去年相比有无差距 |
| 周期环比（本月 vs 上月） | 本月订单环比的增速如何 |
| 同年的周期环比（今年「本月比上月」 vs 去年「本月比上月」） | 今年的增速和去年相比如何 |

落到看板上，把核心指标按「绝对值 / 周同比 / 本月年同比 / 时间进度 / 目标达成」组织，**只对真正需要提醒的指标用条件格式标红黄绿**（如「年同比下降、但增速的年同比却在改善」这种矛盾点），目的是凸显矛盾，而非无脑堆指标。

#### 验证周末效应的两种朴素做法（中性造数）

- 直接按季度算工作日 vs 周末的日均订单差，看差距是否稳定：

  | 季度 | 工作日日均 | 周末日均 | 工作日 vs 周末 |
  |---|---|---|---|
  | Q1 | 978 | 868 | +13% |
  | Q2 | 1,252 | 1,139 | +10% |
  | Q3 | 1,458 | 1,287 | +13% |
  | Q4 | 1,297 | 1,183 | +10% |

- 或用相关性说明二者关系；也可直接用 SQL 把「每周第 1、2…天的日均订单」曲线跑出来描绘周内形态（见文末 SQL）。季节性增长往往一句业务常识即可解释（如旺季环境好、淡季差），这类业务季节性通常很明显。

### 四、第三步：先选对拆解轴——为什么常常不能套「新增/回流/留存」

拿到任一数字（如「这周订单 1000 单」），自然要问两件事：**①多还是少？ ②多/少是什么因素引起的？** 答完就是经典面试题：「这个指标比之前涨/降了多少？为什么？」但**拆解的第一步是选对轴**，而不是机械套用户分层。

在有会员机制、需求高度规律化的业务里，用户其实**已经按付费形态自动分层了**，再切新增/回流/留存几乎没有增量信息。用中性造数感受：

| 用户类型 | 新增 | 回流 | 留存 | 合计 |
|---|---|---|---|---|
| 会员/套餐用户 | 5% | 25% | 70% | 100% |
| 单次付费用户 | 5% | 70% | 25% | 100% |

会员/套餐用户绝大多数是留存，单次付费用户绝大多数是「回流」（没有规律、想起来才用）。这种情况下再机械拆三类，只是公式上的搬运，解释不了波动。更进一步，**会员/套餐用户本身具有业务韧性、很难导致订单下降**：

- 画像是核心、规律的高频用户，几乎不会离开——真要少下单，多半是「这次需求被别处满足了」，而不是「不再需要了」。
- 产品机制兜底：买了套餐就有一段时间的权益，只要还在有效期、当下又恰好满足了需求，用户就倾向继续用、甚至续费。
- 结论：只盯「会员/套餐用户有没有流失」常陷入死胡同——订单降了、会员订单降了，但会员用户没有流失迹象，他只是今天没下单而已，分析没有下文。这正说明要换轴。

正确的拆法是切**订单类型 × 地理区域**，因为不同订单类型已预示不同人群、不同区域预示不同价值。

### 五、沿「订单类型 × 核心区」两条轴做链式拆解

一张诊断表的骨架（中性造数，看的是结构而非数字。此处以某到家服务平台的「会员订单/单次订单」「连续包月/月度套餐」「核心商圈/非核心」为例）：

| 指标 | W1 | W2 | W3 | W4 | 环比W2 | 环比W3 | 环比W4 |
|---|---|---|---|---|---|---|---|
| 总订单 | 1,000 | 1,100 | 1,200 | 900 | +10% | +9% | -25% |
| 会员订单 | 710 | 715 | 864 | 612 | +1% | +21% | -29% |
| 月度套餐订单 | 440 | 379 | 562 | 416 | -14% | +48% | -26% |
| 月度套餐人均订单 | 12 | 13 | 13 | 12 | +3% | +6% | -9% |
| 核心区月度套餐订单 | 379 | 345 | 494 | 375 | -9% | +43% | -24% |
| 连续包月订单 | 270 | 336 | 302 | 196 | +25% | -10% | -35% |
| 核心区连续包月订单 | 183 | 239 | 187 | 143 | +30% | -21% | -24% |
| 单次付费订单 | 310 | 400 | 324 | 273 | +29% | -19% | -16% |

两条轴的意义：

- **订单类型轴**：会员和单次是两类人；会员内部，月度套餐和连续包月又是两类人。若定位到「会员在降、且连续包月降幅最大」，说明**更忠实的核心用户在减少订单**，比月度套餐降更值得警惕，要进一步定位到具体省区/城市。
- **核心区轴**：核心区是承载大多数订单的核心地段。**核心区订单下降意味着核心地段丢失，比非核心区敏感得多**——它直接决定你要投入多大力度、多急的资源去解决。

链式拆解的通用套路（以订单下降为例，也可换成「供给/履约侧」起手）：

```
订单↓
 └→ 是哪类订单在↓（会员 / 单次；月度 / 连续包月）？
      └→ 在不在核心区？核心区占比有没有变？
           └→ 对比去年也↓？ 环比上月也↓？ 距目标还有差距？
                └→ 分城市看：哪些城市在↓？
                     └→ 这些细分维度同样做「同比 / 环比 / 对目标」三重确认
```

要点：这是描述性 / 图形 / 分解分析，本质就是繁琐的对比拆解，给出约八成可信度的结论，剩下靠业务确认。**别在这件事上耗太久**——一般看一两小时数据就该结合业务沟通做简单汇报，要正式报告再额外花半天；觉得很麻烦很困难，往往说明方向或方法有问题。

### 六、两段式钻取：先定位「在哪」，再看「那里的人怎么了」

确定「是哪类订单、在不在核心区」之后，做两段钻取回答「问题在哪 + 存在多久」。

**第一段：若是连续包月在降，是哪个城市贡献了下降。** 把连续包月订单按城市拆，并看各城市的核心区占比随周变化：

| 连续包月订单 | W1 | W2 | W3 | W4 | 核心区占比 W1–W4 |
|---|---|---|---|---|---|
| 城市A | … | … | … | … | … |
| 城市B | … | … | … | … | … |
| 城市C | … | … | … | … | … |

**第二段：定位到城市后，看这群用户近期的健康度。** 关键看随周走势：

- 连续包月订单 / 连续包月人均订单 / 连续包月单均价格
- **用户平均续约期数**（最能反映长期忠诚度的变化）
- **用户退款率**

把两段叠起来——一个是「拆解逻辑（往下切类型与区域）」，一个是「钻取逻辑（盯住一类指标看深）」——就回答了「问题在哪里、有多久」，数据和业务场景由此真正串联。

### 七、串联后的结论该怎么说（示意）

一次订单下降，可以这样讲：

1. 订单下降主要是会员在降，且**连续包月比月度套餐降得更快**；
2. **月度套餐的下降来自外部环境（如天气）**——其近 30 天人均下单频次没大波动、核心区订单稳定；
3. **连续包月降幅更大**，部分城市丢失了核心区的连续包月订单，可能是供给减少或竞争加剧，需与城市侧对齐；
4. 连续包月**单均价格没明显波动，但用户平均续约期数开始下降、且集中在上述城市**——存在长期下降风险。

把这四点说全，才算把一次订单波动的「用户侧原因」讲清楚。它只回答「问题在哪」；「如何解决」属于另一层（往往要继续往供给/履约侧深挖）。

### 八、数据能告诉你「哪里变了」，但讲不出「为什么变」

数据拆解能定位到「哪些城市、甚至哪个网格不产单了」，但它没有感情，回答不了背后的业务因。真正的归因要从「数据拆解」延伸到「业务假设」：顺着指标体系铺开，对每个环节预设业务问题，例如围绕「本月订单下降」：

- **用户侧**：会员/套餐用户是否在减少、单次付费的人均单价与频次是否异常、付费/续费比例是否变化。
- **供给/履约侧**：有效供给占比是否下降、资产/网点老化导致单产下降、闲置资源在区域/时长上的分布、调度与维护是否出问题、库存/流转是否异常。
- **成本侧**：各地投入量/价是否变化、订单下降时是否需同步压成本、是否需要中台干预。
- **市场侧**：竞对动作是否变动，是否需要据此修订口径。

这套清单不必每次全跑一遍。高效做法是：**长期熟悉公司内部本就存在的那些问题，等问题被摆上台面时迅速把分析「端上去」**；让数据涨得更好的分析，同样是好分析。

### 九、为什么这里基本轮不到模型（经验观点）

- 一个精妙的分析 = 统计学知识 + 业务知识 + 逻辑推理，**不是沿某一条技能树往上爬就能解决**。初学者易陷入「简单方法不行就上更复杂方法」的误区。
- 回归、ARIMA 及其他时序模型的最终用途多是测相关性和做预测，这两个需求**在金融量化之外很难产生生产力**：就算把明天订单预测得很准，对策略也没多大用；就算算出「多一份供给多两单」，也不能据此线性外推，因为没纳入成本、运营、产品、竞争等真实复杂度。
- 而且即便只看「订单」这一个指标，从天气、用户、供给、地理等角度就能衍生出几乎所有问题并尝试归因——**到这一步都还动不了模型**。逻辑链是：用户不留存，是因为他不是会员用户；他不是会员用户，是因为他选择不买；不买之前，可能是使用次数、登录次数先减少……没有下钻到真正的业务原因，模型根本没到能发挥作用的位置。在那之前，靠的都是分析师去拆指标、还原背后逻辑。

### 十、底层心法

这套东西听懂不难，难只难在「开始变复杂了」。它恰恰说明：数据分析的核心不是工具、技巧或某个方法，而是**信息输入是否充分、能否把信息串联成逻辑**。业务细节至关重要——不仅外部环境会影响订单，不同产品逻辑与功能也会塑造不同的用户画像。具体怎么影响，需要你认真熟悉业务、站在用户角度想问题，**先把它落成指标口径，最后才是分析数据**。

### 相关 SQL（顺手熟悉时间函数）

```sql
-- 按月统计订单
SELECT DATE_FORMAT(date, 'yyyy-MM') AS month, SUM(order_num) AS total_orders
FROM table_name
GROUP BY DATE_FORMAT(date, 'yyyy-MM');

-- 按自然周统计订单
SELECT YEAR(date) AS year, WEEK(date) AS week, SUM(order_num) AS total_orders
FROM table_name
GROUP BY YEAR(date), WEEK(date);

-- 按工作日 / 周末统计订单
SELECT CASE WHEN DAYOFWEEK(date) IN (2,3,4,5,6) THEN '工作日' ELSE '周末' END AS day_type,
       SUM(order_num) AS total_orders
FROM table_name
GROUP BY CASE WHEN DAYOFWEEK(date) IN (2,3,4,5,6) THEN '工作日' ELSE '周末' END;

-- 每个季度内，按星期几计算平均日订单曲线（描绘周内形态）
SELECT quarter, day_of_week,
       COUNT(order_id) / COUNT(DISTINCT week_of_year) AS avg_daily_orders
FROM (
  SELECT order_id,
         QUARTER(order_date)    AS quarter,
         DAYOFWEEK(order_date)  AS day_of_week,
         WEEKOFYEAR(order_date) AS week_of_year
  FROM orders
) t
GROUP BY quarter, day_of_week
ORDER BY quarter, day_of_week;

-- 订单年同比（本月 vs 去年同月）
SELECT t1.month, t1.total_orders, t2.total_orders AS last_year_orders,
       ROUND((t1.total_orders - t2.total_orders) / t2.total_orders * 100, 2) AS yoy_growth_rate
FROM (
  SELECT DATE_FORMAT(date, 'yyyy-MM') AS month, SUM(order_num) AS total_orders
  FROM table_name GROUP BY DATE_FORMAT(date, 'yyyy-MM')
) t1
LEFT JOIN (
  SELECT DATE_FORMAT(DATE_ADD(date, INTERVAL -1 YEAR), 'yyyy-MM') AS month,
         SUM(order_num) AS total_orders
  FROM table_name GROUP BY DATE_FORMAT(DATE_ADD(date, INTERVAL -1 YEAR), 'yyyy-MM')
) t2
ON t1.month = t2.month;

-- 按周统计各类订单（会员/单次、套餐细分、核心区细分）
-- 注意：年-周应基于订单自身的下单时间字段（如 from_unixtime(o.create_time)）计算并 GROUP BY；
-- 切勿用 unix_timestamp() 取当前时间，否则会把所有订单都归到「跑数当周」，是常见口径错误。
SELECT
  o.year_week,
  COUNT(CASE WHEN o.order_status = 1 THEN 1 END) AS single_pay_orders,   -- 单次付费
  COUNT(CASE WHEN o.order_status = 2 THEN 1 END) AS member_orders,       -- 会员/套餐
  COUNT(DISTINCT CASE WHEN p.plan_name LIKE '%月度套餐%'  THEN o.order_id END) AS monthly_plan_orders,
  COUNT(DISTINCT CASE WHEN p.plan_name LIKE '连续包月%'   THEN o.order_id END) AS continuous_monthly_orders,
  COUNT(DISTINCT CASE WHEN g.location_type = '核心'      THEN o.order_id END) AS core_location_orders
FROM orders_table o
JOIN plans_table   p ON o.plan_type_id = p.plan_type_id
JOIN geo_info      g ON o.latitude = g.latitude AND o.longitude = g.longitude
GROUP BY o.year_week
ORDER BY o.year_week;
```

### 常见坑 / 边界

- **填充/插值要用业务上「本应相似」的样本**（同为周一、相近天气），别图省事用相邻日做移动平均，否则把业务规律填没了。
- **经过删除、填充、转换的数据已不等于真实业务量**，对外解释结论时要明确说明处理过程。
- **同比优先于环比来消季节性**，但务必核对春节等节点在两年里是否对齐。
- **不要在有会员机制的规律性业务里机械套新增/回流/留存**——用户已按付费形态自动分层，这样切只是搬运公式。
- **会员/套餐线天然有韧性**：有效期 + 偶发「刚好被满足」会反复留住用户，只盯「会员是否流失」容易把分析做进死胡同；要切到订单类型与区域。
- **核心区比非核心区敏感得多**：同样的下降，核心区丢失要用更急的资源去解决。
- **续约期数 / 退款率是会员用户健康度的早期信号**：单均价格稳但续约期数掉，往往预示长期风险。
- **链式拆解的每一层都要做「同比/环比/对目标」三重确认**，否则容易把正常季节性当成异动。
- **不要为追求精确预测而上复杂时序模型**；本场景里描述性 + 对比 + 拆解才是主力，模型最多作通识或辅助参考。
- **示例中的百分比、留存率、单均价格均为经验示意**，必须用本业务实测口径替换后再下结论。
