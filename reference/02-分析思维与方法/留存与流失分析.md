## 正文

### 为什么要先定义流失
"流失用户"不是天然存在的，它是一个需要人为划线的口径："连续多少天不登录/不活跃，就判定为流失"。这个 N 值定得对不对，直接决定后续流失分析、预警模型、召回策略是否对得上业务。拍脑袋定 N（比如"7 天没来就算流失"）很可能与产品真实使用节奏不符——低频工具型产品 7 天不用很正常，高频社交产品 3 天不来可能就已经凉了。所以需要用数据把这个周期估出来。下面两种方法都能给出这个 N，且对活跃、付费、复购等各类流失通用。

### 方法一：拐点法
核心直觉是：随着"未活跃天数 N"拉长，用户回来的概率会越来越低，最终趋于一个稳定的低值；这个由下降转为平缓的转折点（拐点）对应的天数，就是流失周期。过了拐点，"该走的基本都走了，该留的基本都留了"。

实操上有两种等价视角：
- **回流率视角（向前回溯）**：以统计日为基准往回看。N 日回流率 = 在统计日之前连续 N 天未登录、但在统计日又回来的用户数，除以连续 N 天未登录的用户总数。逐一计算 N=1,2,3… 的回流率，画成曲线找拐点。
- **留存率视角（向后观察）**：以某天活跃用户为基准往后看。N 日留存率 = 这批用户在第 N 天仍活跃的人数，除以基准日的活跃总数。回流率算起来较绕，工作中常用留存率近似代替来找拐点，结论方向一致。

读图方法：把 N 日回流率（或留存率）随 N 的变化画出来，曲线明显走平的那个 N 就是流失周期。

### 方法二：分位数法
核心做法是统计"用户两次活跃之间的时间间隔"分布。把所有用户的活跃间隔都算出来，看累计占比，取累计达到 90%（或业务认可的高分位）的那个间隔天数，作为流失周期。含义是：九成的"正常回访"都发生在这个间隔之内，超过它还不回来的，大概率是真的走了。

### 中性示例
某 App 计算各 N 值的回流率，发现 N 从 1 增到约 20 天时曲线持续下降，20 天之后基本走平。则判定流失周期 ≈ 20 天，即"连续 20 天未登录"即视为流失。用分位数法交叉验证：统计全体用户活跃间隔，累计 90% 落在 18~21 天附近，与拐点结论吻合，可放心采用 20 天作为口径。

### 常见坑 / 边界
- **拐点不总是肉眼可辨**。有些产品曲线下降很平滑、没有明显折点，这时分位数法更稳；两法互相验证最佳。
- **分位选多少要看业务**。90% 是常用默认，但容忍度高/低的产品可上调或下调；分位越高，流失判定越宽松（周期越长）。
- **不同人群周期不同**。新老用户、不同渠道、付费/非付费的流失周期可能差异很大，必要时分群各算各的，别用一个全局 N 套所有人。
- **口径要随产品形态复核**。产品改版、使用频率变化后，原来的 N 可能失效，需定期重估。
- **流失≠卸载**。这里定义的是"行为意义上的流失"（长期不活跃），与是否卸载是两回事，别混用。
