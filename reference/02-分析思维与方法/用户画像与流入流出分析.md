## 正文

### 一、用户流入流出分析

#### 1. 它是什么，不是什么
严格讲，「用户流入流出」不算一个独立的分析方法，更像是把已有的通用分析套路（漏斗、留存、生命周期、用户调研等）拼装起来，用于解决一类特定问题：用户从哪来、又在什么时候走。它本质上是一个面向具体业务场景的专项分析，而不是一条能直接套用的公式。

- **流入**：用户被吸引、开始使用产品或服务的过程。常见获取手段包括广告投放、内容营销、搜索 / 应用商店优化、渠道合作等。分析流入主要回答：哪些渠道、哪些策略带来的用户最多、最有效。
- **流出（流失）**：用户停止使用产品的行为。可分两类——一类是用户主动放弃（不再需要、体验不好），另一类是被竞品或其他外部原因带走。分析流出主要回答：用户为什么走、什么时候走、如何降低流失。

#### 2. 关键前提：先懂业务，再看数据
做这类分析最容易踩的坑，是直接埋头跑数据、得出「用户最后一次活跃在某天」这种结论——这种基于杂乱明细的、单纯按「最后一次行为」推导的判断，既无法还原真实流失原因，结论也不会被业务采纳。

正确的做法是先把业务场景想清楚：
- 用户当初从哪个渠道、为什么进来；
- 用户通常在什么阶段、出于什么动机离开；
- 在动手取数之前，最好先做用户调研 / 访谈，理解流失的变动、节奏、生命周期阶段。

如果脱离业务场景，光靠一个公式去算「流失率」，对业务几乎没有帮助。

#### 3. 一个推演的例子（中性造数说明思路）
设想一个「内容平台」类业务，要分析某来源渠道用户的流失：
- 先做假设：该渠道用户的内容消费次数在近期明显下降，是不是因为这批人本来就不是平台的目标受众？系统数据也显示该渠道始终有稳定的新增导入。那么这部分「流失」很可能是被竞争对手或其他同类内容产品分走，又或者用户本就没有持续需求（比如只是被某条爆款内容临时吸引进来看一眼）。
- 再细看用户的付费 / 订阅行为：如果用户从没开过会员、互动 / 收藏等深度行为偏少，且只在某个单一话题下停留过，那很可能是被单条内容引流来的过客，看完即走——这类「流失」应当从分析中剔除，把重点放在找出真正值得干预的流失人群上。

要点：**想清楚业务，分析逻辑才有意义；杂乱明细只能给方向，不能直接当结论。**

#### 4. 把流失原因建模（指标筛选思路）
进一步可以把「流失前的若干关键行为」作为自变量，去回归 / 建模拟合流失结果，再上线监控、配置预警，从源头介入流失。

筛选指标时不能拍脑袋，要让指标和业务动作能对应起来。举例：
- 可选的候选指标有【浏览行为】【浏览时长】【浏览类型】，以及最终的流失标签；
- 假设建模后发现「浏览时长越长，越不容易流失」，那么运营就要去经营、提升用户的浏览时长（实际中往往多个指标一起影响，这里只是举例）；
- 反之，若把【浏览行为】【活动参与】【产品功能使用】【付费行为】这些代表不同组织架构、不同业务动作的指标一起放进去，发现最后和流失强相关的是「产品功能使用」，就能指向「流失主因是产品功能问题，而非内容不够好」这类有抓手的结论（这时还要辅以其他描述性指标交叉验证，确认确实如此）。

#### 5. 这件事更深一层的意义
做流入流出分析时，要多想一层分析师的职能：很多时候它并不是为了「证明哪个指标更重要」，而是借由指出「公司当前的运营动作里存在什么问题」，来引导资源往该投的地方配置。**指标只是抓手，引导资源配置才是目的。**

### 二、用户画像构建思路

#### 1. 定位：它是工程，不是分析方法
用户画像本质上更接近一项「开发工程」，是公司发展到一定阶段、为了精细化运营、降低运营成本而建设的能力。如果在业务还很早、铺设条件不成熟时就强上画像，画像往往只是「被深度研究的指标」，不能真正带来业务提升。要构建画像，最好从分析视角先想清楚「我们为什么要用、怎么用」。

#### 2. 本质：一连串描述用户的指标
画像的核心，其实就是一组（可能几百个）描述用户的指标，每一个指标都是一个标签 / 指标仓库里的条目。它有两面价值：
- **对内（运营 / 算法）**：在企业内部精细化运营场景里用，比如圈选「近 N 天消费 X 元以上」的人群发优惠券；
- **对外（产品 / 用户）**：把画像透出给用户本人看，用更好引导其行为。

行业里常见的成熟画像体系，例如某些电商平台的人群分层模型、某些内容平台的人群分层模型等，都是用画像 / 标签把人群结构化的一部分。

#### 3. 画像本质是「为目的服务的指标体系」
构建画像，分析师真正要做的是搭一套用户指标体系。这要求先理解业务、用户需求、以及被研究人群的聚集特征。

举一个标签定义的例子（说明口径需要业务约束）：
- 想做「流失用户」这个标签，「什么样的用户算流失」需要业务来定边界——在某些有刚需的领域，用户日活跃率很高，那么「留存很差的情况下」可能 30 天没登录就算流失；但这种指标不能拿到所有产品上套用。
- 换个产品，可能「永不登录」「不付费」「只登录一次但长期不再登录、且后续大概率不会再登」才算流失。**同一个「流失」标签，在不同业务里口径完全不同**，要先把口径约定清楚，再用这种方式逐个建标签，最后汇成数百个标签的画像。

#### 4. 站在分析师视角，一个用户画像通常包含三层
1. **基础属性**：性别、年龄、注册时间等——这些是埋点之初就该采集的；
2. **基于业务的行为数据**：某功能的使用次数、内容观看次数、登录周期、流失情况等；
3. **基于数据分析结论的综合类指标**：例如用多个指标判断出来的「流失用户 / 留存用户」、刻画用户特点的标签、业务长期需要的用户类型分层等。

#### 5. 分析师在画像里实际做什么
主要是通过 SQL 把上面这套逻辑落到一张宽表里，再基于宽表自己产出分析结论。比如：
- 监测每天的流失用户波动是怎样的、原因是什么；
- 另一边推动产品 / 算法把画像对接到系统中，让运营、算法能直接用这些标签给用户发券、做推荐、建模等，画像建到此才算闭环。

> 画像「更像一个工程，而非一次性项目」。它的雏形可能只是一张用户宽表，只有在长期被反复使用的情况下才真正有价值，而这种价值更多是「提效」的价值——毕竟用户量少的时候，一张 Excel 表也能解决这些问题。建画像的前提是规模和复用场景都到位。

### 常见坑 / 边界
- **脱离业务跑数据**：流入流出分析最大的坑就是不懂为什么流失，单看「最后活跃时间」之类明细就下结论，业务不会买账。
- **指标拍脑袋**：建模 / 选标签时要让指标对应得上具体业务动作，否则相关性再强也无法转成行动。
- **过早建画像**：规模小、复用场景少时强上画像投入产出比低，Excel 即可替代；画像是「提效工具」，没有持续复用就没有价值。
- **标签口径不统一**：同一个标签（如「流失」）在不同业务口径迥异，必须先由业务约定边界再开发。
