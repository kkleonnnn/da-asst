## 正文

### 一、为什么要做离群点挖掘

业务里最常被分析的是主流链路和高频场景，但"发生概率最高"不等于"问题价值最高"。一些影响体验的问题（误触、错误跳转、广告过度打扰等）往往集中在少数异常用户身上，正向漏斗分析很难覆盖到。离群点挖掘提供了另一条路径：先用算法把"和大多数明显不同"的数据点挑出来，再回到业务里查这些点背后藏着什么问题。

离群点（Outlier）指数据集中与大多数样本显著偏离的点，成因可能是录入/测量误差，也可能是真实存在的异常情况——后者正是分析想抓的目标。

### 二、常见离群算法及选型

按思路可分为四类，核心区别在于"靠什么判断一个点是否异常"以及适用的维度数：

| 思路 | 代表算法 | 原理一句话 | 适用 |
|---|---|---|---|
| 统计 | Z-score | 点到均值的标准差距离超过阈值（常用3）即异常 | 单维、近正态分布；如耗时指标的日常监控、防劣化 |
| 统计 | IQR（四分位距） | 落在 Q1−1.5·IQR 之外或 Q3+1.5·IQR 之外即异常（IQR=Q3−Q1） | 单维、对分布要求低、计算简单 |
| 距离 | K-NN | 点到其K个最近邻的平均距离明显偏大即异常 | 非球形分布；高维时计算成本高，K值敏感 |
| 密度 | DBSCAN | 按密度聚类，落在低密度区、不属于任何簇的点视为噪声/离群 | 无需预设簇数，能发现任意形状簇；对半径ε、最小点数MinPts敏感 |
| 模型 | Isolation Forest（孤立森林） | 随机选特征和切分值建树，异常点更容易被早早"孤立"出来 | 大规模、高维数据表现好；模型可解释性一般，需调参 |
| 密度 | LOF（局部离群因子） | 比较一个点与其邻居的局部密度，密度显著偏低即异常 | 适合密度不均的数据；高维大数据计算复杂度高 |

**选型经验**：
- 单维度指标监控（如耗时、卡顿）→ 统计类（Z-score / IQR）足够；
- 多维度、交互行为类的异常用户定位 → 用密度类（DBSCAN）或模型类（Isolation Forest）这类能处理高维的方法。

### 三、多维场景下的实现流程（三步）

以"交互体验中的多维异常用户定位"为例，流程是 **特征集构建 → 离群点检测 → 离群点分析归因**。

**1. 特征集构建**

选特征时遵循六条原则：
- **代表性**：能代表关键属性/行为，如性能体验场景用耗时、卡顿率；
- **可区分**：能把正常和异常行为分开，如内容场景用完播率；
- **简洁、可解释**：少而信息量大、业务上讲得通，如用"点击率"而非"曝光数+点击数"两个原始量；
- **完整可取**：数据要齐、口径一致，缺失/错值多会让特征失效；
- **低相关**：特征间尽量不相关，避免冗余和维度灾难。

落地手段：用相关分析剔冗余、用方差筛选保留可区分度；必要时用 PCA / t-SNE / UMAP 降维保证简洁；并结合业务专家经验判断哪些行为才真正代表"异常体验"。

**2. 离群点检测（以 DBSCAN 为例）**

- 定关键参数：邻域半径 ε（密度范围）、核心点所需的最小邻域点数 MinPts。ε 常用 K-distance 图来定——算每个点到第k近邻的距离再排序，取曲线"拐点"处的值。
- 检测过程：初始化所有点为"未访问"；随机选未访问点，算其 ε 半径内的邻域点数；若 ≥ MinPts 则标为核心点并以它为中心扩簇，把邻域内的点纳入；对新纳入的点递归扩展，遇边界点停止；重复直到全部访问完毕；最终未归入任何簇的点即噪声/离群点。

**3. 离群点分析与归因**

把识别出的离群用户单独抽出来细查，确认背后的体验问题并给改进建议。常用归因维度：
- **用户细查**：看其行为轨迹（操作顺序在哪一步出问题）、关键触点表现；
- **竞品走查**：对比竞品同场景的交互/激励策略，看自己有没有未满足的习惯（如别家短视频长按倍速，自家是否也有该行为却没支持）；
- **维度对比**：离群用户 vs 正常用户在关键特征（使用时长、签到率、消费次数等）上的差异；
- **用研**：访谈、问卷补充定性反馈，了解真实感受。

### 四、应用示例：内容型App的激励ROI异常排查

**背景**：内容型产品的用户增长（UG）业务常用激励任务达成两个目标——签到类任务拉留存/活跃，内容消费类任务（阅读、听书、短剧）拉人均消费时长。业务目标可概括为"用最少预算撬动最大的活跃度和人均时长提升"。

**要找的两类异常用户**：
- **低ROI用户**：花了较多预算，活跃/时长收益却不大；
- **高潜未覆盖用户**：活跃/时长有提升空间，却没被激励触达到。

**特征设计示意**（中性造数口径，仅示意，非真实数据）：

| 维度 | 指标 | 口径示意 |
|---|---|---|
| 活跃&消费 | 各题材消费次数 | `sum(case when event='consume_xxx' then 1 else 0 end) as xxx_cnt` |
| 活跃&消费 | 各题材消费时长 | `sum(case when event='consume_xxx' then cast(duration as bigint) else 0 end) as xxx_time` |
| 留存 | 次日回访比例 / 近30天活跃天数 | 留存率、活跃天数 |
| 任务感知与完成 | 任务展现次数、展现点击率 | `点击数*1.0 / 展现数 as xxx_click_rate` |

**异常规则归纳**：
- 低ROI ≈ 感知并完成了大量内容消费任务但后续时长未涨 / 完成了大量签到任务但活跃留存未涨；
- 高潜未覆盖 ≈ 时长、活跃仍有提升空间，但对激励任务感知弱或没去完成。

**建模骨架**（伪流程，造数示意）：
1. 读数 → 空值/异常值处理（如用列均值填充、对超过3倍标准差的值做截断）；
2. 归一化（MinMax 或标准化）；
3. 相关性分析剔冗余 → PCA 降维（如保留95%方差）；
4. 用 Isolation Forest 检测离群点（设 contamination，如假设5%为异常）→ 输出每个用户的 Normal/Outlier 标记。

**归因示例**：
- 某"低ROI"用户：当日消费时长高、领了多次补贴，之后时长骤降——细查发现该日处于弱网环境且清晰度仍按高清拉流。建议对弱网用户做清晰度档位降级。
- 某"高潜未覆盖"用户：日均听书时长高、对激励任务有感知但不点击——细查发现推荐的任务题材与其消费偏好不匹配。建议提升任务与用户偏好的匹配度、调整金币任务的题材配比。

### 五、常见坑与边界

- 离群 ≠ 问题：算法挑出来的点可能是数据错误或无害的极端值，必须回到业务里细查才能定性。
- 参数与阈值敏感：DBSCAN 的 ε/MinPts、Isolation Forest 的 contamination、Z-score 的阈值都直接影响结果，需结合分布反复校准。
- 高维下统计类方法失效：单维方法别硬套到多维场景，反之亦然。
- 依赖业务先验：特征选择和归因都建立在对业务、数据有一定理解之上，盲目套模型容易得出无意义的离群群体。
- 定位为补充洞察：这是对主流链路分析的补充而非替代，适合在存量竞争中挖低频高价值问题，属探索性方法，结论需谨慎验证。
