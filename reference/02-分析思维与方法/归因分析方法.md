# 归因分析方法

> 所属板块：02-分析思维与方法
> 类型：方法论 / 框架

## 元信息
- **适用范围**：两类归因场景。其一是「某指标变差、要查清原因并给出可落地方案」的开放式问题拆解（尤其留存 / 流失 / 转化类，也适用于活动复盘、功能效果排查）；其二是「用户在转化前接触多个渠道、渠道间有先后组合关系」的多触点价值分配（如某电商 / 某游戏发行 / 某 App 的拉新投放）。不适用于已知口径只需算个数的取数需求；多触点归因还要求路径并非单一触点、且样本量足以估计转换概率。
- **时效**：整理日 2026-06-25；内容长期有效（属通用分析框架与方法原理）。
- **可信度**：高 —— 5W2H 是成熟提问法、落到数据场景的拆法逻辑清晰可直接套用；马尔可夫链归因有明确的概率论与状态转移逻辑支撑、移除效应计算可复现。

## 正文

> 「归因」要回答的核心是：**一个结果（指标变化、一次转化）该归功 / 归咎于什么。** 它有两条常见路径——一条是把模糊问题层层拆成可分析的子问题、逐个定位原因（问题拆解式归因，代表是 5W2H 闭环）；另一条是在多个并存的触点 / 渠道之间，用数据量化各自的不可替代性、分配价值（数据驱动归因，代表是马尔可夫链 + 移除效应）。前者偏「找原因 + 给方案」，后者偏「分功劳 + 配预算」。两者都只能给出「贡献 / 相关」层面的结论，真正的因果仍要靠实验验证。

---

## 一、问题拆解式归因：5W2H 闭环

### 一句话方法论
5W2H 是一套「用七个问句把一个模糊问题钉死成可分析子问题」的提问框架——What / When / Where / Who / Why 加 How / How much——它本身不产出结论，而是逼你把「留存为什么降了」拆成一串能用数据回答的小问题，再配漏斗、分群等方法逐个下钻，最后形成「问题→定位→推因→方案→验证」的闭环。

### 七个问句怎么落到数据上
不要把 5W2H 当填空题死记，要理解每个问句对应的分析动作：

- **What（是什么问题）**：先把问题量化成具体指标和缺口。例如「新用户次日留存比同类基准低 8 个百分点且仍在走低」，而不是笼统说「留存不好」。这一步决定了后面所有分析的靶子。
- **When / Where（何时、在哪个环节）**：用漏斗分析定位流失发生在哪一步、哪个阶段。这里的「Where」通常指业务路径上的位置，而非地理位置。比如发现八成流失集中在新手引导期，就把镜头推进到引导期的分步漏斗。
- **Who（是谁在流失）**：用分群 / 用户画像找出流失人群的属性——来源渠道、设备档位、新老、地区等。常见结论形如「流失里新手占比畸高」「低配机型流失更重」。
- **Why（为什么）**：综合上面三步的现象，提出对原因的假设。注意这里得到的是「假设」，不是定论。
- **How / How much（怎么改、改到什么程度 / 花多少）**：把假设转成具体的产品或运营调整方案，并估算成本与预期收益。

### 标准套路（五步闭环）
1. 量化问题、锁定靶子指标（What）。
2. 漏斗定位流失环节（When / Where）。
3. 分群刻画流失人群（Who）。
4. 据现象提出原因假设（Why）。
5. 设计调整方案、排期、用 A/B 实验验证假设是否成立、调整是否有效（How / How much）。
最后一步验证完成后回看靶子指标，形成闭环；若无效则回到假设环节重来。

### 中性示例
某休闲游戏次日留存偏低。What：次留比基准低 8pp。Where：分步漏斗显示 80% 流失发生在前几关（即新手引导 + 前几局），其中引导期内某一步骤的单步流失率明显异常。Who：流失用户里新手与低内存机型占比偏高。Why：推测引导步骤设计不友好、低配机型卡顿。How：简化该引导步骤、为低配机型降画质；随后用 A/B 实验对比新旧引导的次留差异。

### 常见坑 / 边界
- **别把 Why 当成结论直接交付**。数据只能看到「现象相关」，真正的因果要靠实验验证；跳过验证就给方案，很容易把相关当因果。
- **5W2H 是骨架不是全部**。它负责保证拆解不漏维度，真正下钻还得靠漏斗、分群、对比等方法填进去；只列七个问句而不接分析方法，等于纸上谈兵。
- **顺序可按业务调整**。不同业务里 When / Where / Who 的先后并不固定，关键是「先定位再推因后验证」这条主线不能乱。
- **方案要落在可调可控处**。能改的是产品和运营侧（内因），对不可控的外部因素，5W2H 的产出更多是「理解」而非「行动」。

## 二、数据驱动归因：马尔可夫链与移除效应

### 一句话方法论
把用户从「第一次接触」到「最终转化 / 未转化」的全过程建模成一条状态链，用各状态间的转换概率算出整体转化率，再逐一「拿掉」某个渠道看转化率掉多少——掉得越多，说明这个渠道越重要。

### 为什么需要它
传统归因（首次接触、末次接触、线性平均、时间衰减、U 型）本质都是**人为预设的分配规则**：要么把功劳全给第一个触点，要么全给最后一个，要么平均分。它们简单，但都无法回答「到底哪个触点真正不可替代」。当渠道多、组合复杂时，预设规则容易把高价值触点和陪跑触点混为一谈。数据驱动归因的思路是：不预设分配规则，而是让真实路径数据自己说话。

### 怎么做（五步）
1. **记录每条转化路径**。给每个用户的旅程加一个统一的起点状态「开始」，终点状态固定为「转化」或「未转化」两者之一。中间是用户依次接触的渠道。例如一条路径是 `开始 → 渠道A → 渠道B → 转化`。
2. **统计状态间的转换次数与转换概率**。对每一个状态，数清它流向各个后续状态的次数，再除以它流出的总次数，得到转换概率。校验口径：**从任一状态出发，流向所有可能后续状态的概率之和必须等于 1**。
3. **算整体转化率（整体激活率）**。把从「开始」能走到「转化」的每一条路径找出来，路径上各段转换概率连乘得到该路径的转化贡献，再把所有通往「转化」的路径相加。
4. **算每个渠道的移除效应系数**。公式：`移除效应系数 = 1 −（移除该渠道后的整体转化率 ÷ 原整体转化率）`。做法是把该渠道及其相关连边从链中删掉，重新计算剩余路径能达成的转化率。
5. **按移除效应系数分配价值**。某渠道的最终贡献占比 = 它的移除效应系数 ÷ 所有渠道移除效应系数之和。据此分配预算与 ROI 核算。

### 中性示例（造数）
设两个渠道 A、B，整理后得到转换概率：
- 开始 →（A 概率 0.8，B 概率 0.2）
- A →（转化 0.25，B 0.5，未转化 0.25）
- B →（转化 0.67，未转化 0.33）

通往「转化」的路径有三条，连乘后相加：
`0.8×0.25 + 0.8×0.5×0.67 + 0.2×0.67 ≈ 0.2 + 0.27 + 0.13 = 0.6`，即整体转化率 60%。

移除 A 后只剩 `开始→B→转化`：`0.2×0.67 ≈ 0.13`，则 A 的移除效应 = `1 − 0.13/0.6 ≈ 0.78`。
移除 B 后只剩 `开始→A→转化`：`0.8×0.25 = 0.2`，则 B 的移除效应 = `1 − 0.2/0.6 ≈ 0.67`。

结论：A 比 B 更不可替代——理论上没有 A 会损失约 78% 的转化。注意这两个系数不必相加为 1，它们各自衡量「缺了它系统会塌多少」，分配价值时再做归一化。

### 移除效应到底是什么
就是**「抽掉某个状态后，从开始到成功转化的所有路径概率之和下降了多少」**。下降得越多，这个触点对整个转化系统越关键。它衡量的是「不可替代性」，而不是「经手次数」——这正是它优于末次 / 线性归因的地方。

### 常见坑与边界
- **样本量要够**。转换概率是频率估计，路径稀疏时（某些状态只有一两个样本）概率噪声极大，结论不稳。
- **马尔可夫假设的局限**：模型假设「下一步只取决于当前状态」，忽略更早的历史。真实用户心智有记忆，这是简化。
- **同质化路径**：如果几乎所有用户都走同一条路，移除效应会把功劳极度集中，要结合业务判断是否合理。
- **只回答「贡献」不回答「因果」**：移除效应是观测数据上的反事实近似，不等于做实验得到的因果增量；预算大调整前最好再用实验验证。
- 渠道极多时可先合并低频渠道，避免状态空间爆炸。

## 两种方法怎么选
- 命题是「某指标为什么变差、该怎么改」→ 用 **5W2H 闭环**做问题拆解，定位环节与人群、提假设、做实验。
- 命题是「转化前有多个渠道 / 触点，谁更该拿功劳、预算怎么分」→ 用 **马尔可夫链 + 移除效应**做数据驱动的价值分配。
- 两者并不互斥：5W2H 的 Why / How 环节里，「哪个渠道更值得投」这类子问题就可以用移除效应来量化；而移除效应给出的「贡献排序」，最终也要靠实验去验证因果增量。
