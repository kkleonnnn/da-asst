## 正文

### 一句话方法论

机器学习的门槛在「数学 + 编程 + 业务」三者叠加，能跨过的人少。这类工具的思路是：**在 SQL 之上加一层抽象，把「训练模型」「用模型预测」也写成 SQL 语句**。于是只要会 SQL，就能自助建模和预测——把 AI 的使用权从少数专家手里下放给广大业务人（即所谓「AI 平民化」）。

### 它解决的痛点

数据仓库里数据齐备、懂 SQL 的人也多，但「从数据到模型」这一步卡着一道墙：要会 Python、要懂特征工程和调参、要会把训练好的模型工程化部署。结果是业务专家明明最懂场景，却用不了模型，只能把需求转交给稀缺的算法资源，效率低、迭代慢。把建模动作收进 SQL，就是把这堵墙拆矮，让「会查数的人」直接「会建模」。

### 核心范式：声明式地表达「要什么」，而非「怎么算」

传统写法要自己写循环、定义损失函数、手动迭代——关心的是**怎么算**。SQL 方言的写法只声明**要什么**：

- **训练**：用一句扩展 SQL 表达「拿这段查询的数据，用某个模型、按这些超参，把这几列当特征、那一列当标签，训练好存到某张表」。
- **预测**：再用一句表达「拿这段查询的新数据，套用之前训好的某个模型，把结果写到某张输出表」。

用户只说清「哪份数据、哪个模型、什么参数、谁是特征谁是标签」，至于用哪个底层库、怎么并行、结果怎么存，全由工具去补。**机器学习从「一件特殊的工程」被重新框定为「一次常规的数据变换」**——这正是范式转变的关键。

> 中性示例（仅示意结构，非真实语法）：
> 训练 ≈「SELECT … FROM 用户特征表 训练 某分类模型 用(年龄, 近30天活跃, 历史下单) 作特征 用(是否流失) 作标签 存到 流失模型」；
> 预测 ≈「SELECT … FROM 本周新用户 套用 流失模型 输出到 流失打分表」。

### 工作原理：像一个「SQL→机器学习」的编译器

可以把这层抽象理解成一个翻译器，夹在 SQL 引擎和机器学习引擎之间，大致流程：

1. 用户把扩展 SQL 提交给服务端。
2. 解析器把语句拆成两部分：**标准 SQL 子句**（SELECT / WHERE 等）和**扩展子句**（训练参数、模型类型、特征 / 标签声明等）。
3. 标准部分照常下发给底层数据库去**取数**（得到训练或预测用的数据）。
4. 扩展部分被解析成一个结构化的中间表示，再**转译成调用底层机器学习框架的代码**，去完成特征预处理、按指定超参初始化模型、训练、并把模型序列化保存。
5. 预测同理：SELECT 取来新数据，由「套用哪个已训模型」的子句指定模型，产出结果写回指定表。

这套设计的精髓是**把「声明意图」与「实现细节」解耦**：用户表达意图，系统决定用哪套库、如何执行、结果落到哪。

### 定位与生态

- **不绑死厂商**是这类开源实现的卖点：同时对接多种数据后端（关系库、数仓等）和多种建模框架，而不是只能配某一家云的专有设施。
- 通常内置一批可直接按名字调用的现成模型（常见的分类 / 回归、提升树集成、聚类、面向时序的网络等），也允许贡献者按统一接口添加自定义模型并注册，形成社区共建的模型库。
- 运行形态多为容器化（打包好数据库、交互环境与示例），既可在 Notebook 里交互探索，也可脚本化批量执行。

### 它在「AI 辅助分析」谱系里的位置

本板块其他做法多是「人借对话式 AI / LLM 来辅助分析」；这条范式不同——它不是用自然语言对话，而是**把建模能力直接做进 SQL 这门分析师的母语里**，让自助建模成为数仓上的一等操作。二者目标一致（降低门槛、放大人的产能），路径互补：一个靠自然语言，一个靠声明式 SQL。

### 常见坑 / 边界

- **降的是「使用门槛」，不是「理解门槛」**。语法变简单了，但选错模型、特征有泄漏、标签定义错、不看评估就上线，照样得出错误结论。会写不等于会用，基本的建模常识仍不可少。
- **覆盖的是「常规建模」**。高度定制的结构、前沿精度需求，仍需专业算法工程，这类工具未必够用。
- **可解释与可监控要另配**。声明式很方便，但模型上线后的线上线下一致性、漂移监控、可解释，往往要额外手段补上，工具本身不自动负责。
- **抽象有代价**。封装屏蔽了底层细节，遇到性能瓶颈或诡异结果时，排查反而需要懂底层在做什么——抽象层不是免责层。
