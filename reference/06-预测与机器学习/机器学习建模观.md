## 一、三层模型观：业务模型 → 数据模型 → 函数模型

### 先厘清「模型」到底指什么

「模型」的核心含义是「**参照一定规范与标准而形成的样式**」。理解时抓两点：

1. **参照标准，但不必完整复制。** 既可以是高仿真的复制（如照实物做的飞机模型），也可以是对现实的抽象、简化、抽取、组合（如把一批支付行为整理成一张交互行为数据表——它只截取了一部分、还做了简化）。
2. **落地形态多样。** 可以是物理实体，可以是静态抽象表达（如把用户 ID、年龄、职业组织成一张一维表），也可以是实体之间的关系（如一句「气温升高，冰块融化」描述的就是气温与冰块的关系模型）。

在数据科学里，「规范与标准」常常是复杂的现实世界或某个业务里的客观存在，而落地「样式」基本都是各种数据表达。按使用形态，可把模型分成三层：**业务模型 → 数据模型 → 函数模型**。它们不是并列的三种东西，而更像一个业务问题从「想清楚」到「可计算」的递进。

### 第一层：业务模型——把现实重塑成有序的逻辑

业务模型是把现实世界、复杂事物、具体问题用某种方式重塑后的模型。它的「样式」是抽象的，但常借流程图、结构图等具体形式表现出来。构建它有两个目的：

- 对杂乱的现实逻辑做有序整理，**突出主要矛盾、淡化干扰因素**。整理时难免有简化失真，但要尽量和真实业务保持最大一致，否则价值打折。
- **协同各方、对齐目标。** 复杂业务靠多人协作完成，一个好的业务模型能统一思想、提升效率、降低成本。

在数据科学里，业务模型经常表现为一张**完整的数据处理流程图**。

> 中性示例：某生鲜平台「次日补货量预测」的业务模型，可整理成一条流程——加载货品 / 用户 / 订单数据 → 清洗 → 特征工程 → 预测算法 → 二次筛选 → 输出预测结果。这张流程图就是该任务业务模型的一种表现形式（但业务模型本身是抽象的，流程图只是它的一种画法）。

### 第二层：数据模型——用「组织化的数据」把业务模型落地

业务模型是抽象的，要落地就得有具体载体；在数据科学里，它以**数据模型**的形式落地。数据模型同样源于现实场景，但「样式」更具体——以组织化的数据形式呈现，主要工具是数据库与数据仓库。数据模型一般包含三方面：

- **数据结构**：要描述的数据的类型、内容、性质及彼此联系，是数据模型的基础。
- **数据操作**：对这些数据可做的各项操作（类型与方式）及其推理规则。
- **数据约束**：保证数据在存储与处理中正确、一致、相容的组织规则与依存关系。

以这三者为底层逻辑，常见的数据模型有三种：

- **层次（树状）模型**：树形结构，结构简单清晰、查询高效，但几乎只能表达一对多，难表达多对多。例：组织架构、人事结构、思维导图。
- **网状（图）模型**：有向图结构，解除了父子层级约束，能表达多对多、更复杂的关系，但实体一多关系一杂就很繁琐、增删时耦合大。例：社会关系图谱、物流时空网络。
- **关系型模型**：用表格表示实体及关系，结构简单、操作方便，靠增加一张映射表就能表达多对多，是目前最流行的模型（关系型数据库即其成功应用）；代价是物理化实现时会加重系统负担。

### 第三层：函数模型——用函数表达变量之间的关系

函数模型专门用来表示**实体变量与实体变量之间的变换关系**，用数学函数表达更精准、简洁又不失丰富。它的「样式」是函数，三要素即函数的定义域（输入范围）、值域（输出范围）、对应法则（输入到输出的映射）。

**参数与机器学习的关系**，是这一层最关键的认知：

- 一个函数模型 = **函数形式 + 函数参数**，两者都定了，模型才算最终确定。
- 以 y = e^(ax) 为例：指数关系是「函数形式」；其中的 a 寄生在这个形式上、用来调节映射形态，就是「参数」。
- 确定参数有两条路：① 人工直接指定（或经分析后指定）；② **假定输入与输出的映射已知（有标注），设定一个目标，用某种机制让程序自动算出参数**——这条路就叫机器学习。用数据算参数的函数模型就是机器学习模型，算参数的动态过程叫「训练」。
- 而在算参数的过程中，由人工介入控制计算过程的那些变量（不是被训练出来的，而是事先设定的），叫**超参数**。

> 一句话串起来：业务模型说清「这件事的逻辑长什么样」，数据模型把它「存成可处理的数据」，函数模型把其中变量关系「写成可计算、可训练的函数」——机器学习正是在第三层、用数据把函数的参数解出来。

## 二、三种数据刻画视角：空间、概率、最优化

### 一句话

正式学各种函数模型之前，先认识「**怎么描述数据和数据之间的关系**」。同一份数据可以从三种视角刻画：把样本当**空间里的点**、把样本当**事件的组合（概率）**、把变量关系当**最优化问题**。绝大多数机器学习模型都建立在这三种刻画之一（或组合）之上，认清视角，就更容易掌握具体模型。

### 视角一：空间刻画——把样本当点，关系靠「距离」

这是最常见的视角：把每个样本当成高维空间里的一个点，每个特征是一个相互独立的坐标轴（在不做空间变换前，各维独立存在）。有了空间，就有距离、向量、相似度等几何计算，其中**距离是最基本的手段**。

为什么距离有用？可类比「距离近的事物关联更紧密」这一朴素规律——把样本当空间里的「事物」，距离与相关性的关系就能被利用起来建模。常见距离：**曼哈顿距离 / 欧氏距离 / 闵氏距离**（闵氏距离带参数 p，p=1 即曼哈顿、p=2 即欧氏）。

直接算距离有两个陷阱，要会处理：

1. **量纲不同**：比如「下单金额（元，10~1000）」和「浏览时长（分钟，3~15）」，量纲和分布都不同，直接算距离会让大数量级的特征主导结果。常用**归一化 / 标准化**把各维缩放到同一尺度。
2. **特征相关**：即便统一了量纲，如果「下单数量」和「在线时长」本身高度相关，一个用户在标准化后某维高出 1 个单位，由于相关性，实际距离差可能远超「单维 1 单位」的影响，使距离不再客观。解法是把空间或样本做变换，让变换后各维相互独立：
   - **PCA 变换**：按各维分布与相关性做线性变换，使各维独立（顺带去掉量纲影响，不必再单独统一量纲）；衡量距离时可保留全部维度（不降维）。
   - **马氏距离**：表示样本间的协方差距离，天然与量纲无关，并排除特征间相关性的干扰。

> 注意：马氏距离或去量纲距离并非「一定更好」，方法好不好要靠实际业务检验。
> 以空间刻画为基础的模型很多：监督学习里的最近邻、支持向量机、决策树（空间「切蛋糕」），无监督里的 K-Means、DBSCAN、层次聚类，半监督里的标签传播等。

### 视角二：概率刻画——把样本当「事件的组合」

与「样本是一个点」不同，概率视角把每个样本看成**若干事件的组合（复合事件）**，而每个特征就是构成样本的最小事件。为什么不把整条样本当成单一事件？因为那样太稀疏（要所有特征全一致概率才升高）；把它拆成「每个特征事件」更合适——每个特征值都可视为该样本在这一特征上、相对总体的发生概率。

> 中性示例：某日 1000 笔订单中女性占 70%、男性占 30%。某笔订单顾客是男性，就把它看作一个「性别为男的事件」，发生概率 30%——这就是把「性别」当事件看待。

要用好概率刻画，必须掌握两类工具：

- **联合概率与联合概率分布**：复合事件发生 = 其组成事件**同时**发生。注意 P(A,B) 不一定等于 P(A)·P(B)——只有当两特征相互独立时乘积才成立。把各特征所有取值组合枚举出来、连同各自概率，就是联合概率分布。连续特征可先离散化，或直接用概率密度函数处理；实际研究时不必把所有特征都「联合」，按业务目标只联合感兴趣的部分即可。
- **条件概率**：在某事件已发生的前提下另一事件的概率，写作 P(B|A)，竖线右侧是条件项（即分母）。它与联合概率有固定关系：P(A,B) = P(A)·P(B|A) = P(B)·P(A|B)。

把**标注**引入条件概率，业务含义会非常强。例如给每笔订单加「近一月是否再次购物」的标注：

- P(再次购物 | 女性)：女性用户近一月再次购物的概率——**可直接用于决策判断**；
- P(女性 | 再次购物)：再次购物的用户里女性的占比——**可帮助寻找显著特征**。

> 以概率刻画为基础的模型也不少：朴素贝叶斯、决策树（既可空间刻画也可概率刻画）、关联模型等。

### 视角三：最优化刻画——描述的是「数据之间的关系」

前两种刻画数据本身，最优化视角刻画的是**数据之间的关系**：在一定约束下，用某种方法让目标达到最优。三个要素必须考虑：

1. **目标（目标函数）**：反映特征之间关系的一个表达式（不是等式也不是不等式），通常关注它的最小 / 最大值。例：用当前温度、湿度、降雨量、云量预测一小时后气温，若用线性回归，目标就是让 (预测值 − 真实值)² 最小（平方便于求导）。
2. **约束**：受定义域或额外条件限制，常以等式 / 不等式存在。如降雨量、云量不能为负，这就是约束。
3. **求解方法**：
   - 简单、可符号推导的：直接求导找极值点；有约束时用拉格朗日乘数法等；牛顿法、共轭梯度法也属此类。
   - 复杂或强非线性的：靠**数值计算**。最通用的是**梯度下降法**——把目标函数想象成一座大雾弥漫的山，人只看得见周围一小片，于是一直朝「下山」方向走，总能走到一个极值点；其中**步长**很关键：太小收敛慢，太大会在极值点附近来回「跨」而不收敛。许多求最值思路都是它的变体（如发现一路下坡就加速冲一冲，类似动量法）。
   - 简单线性形态（如线性规划）：可用**单纯形法**，在 N 维空间用 N+1 个顶点构成的单纯形逐步迭代逼近最优。

### 怎么用这三种视角

拿到一个模型先问：**它把样本当点、当事件，还是把关系当最优化问题？** 例如理解 K-Means 走空间刻画、朴素贝叶斯走概率刻画、线性回归走最优化刻画，再去看具体公式就有了抓手。很多模型是组合视角（如决策树兼具空间与概率），这也解释了它们为何灵活。

## 三、归纳偏置与模型选择

### 一句话

任何模型在「数据没明说的地方」都会按某个**先验假设**来下判断，这个假设叫**归纳偏置**。模型好不好用，很大程度上取决于它内置的归纳偏置是否贴合业务现实。所以**选模型 = 选一套和你业务匹配的假设**；而当数据不足以判断该信哪个假设时，用**奥卡姆剃刀**（如无必要、勿增实体）选最简单的那个。

### 什么是归纳偏置：让「学不动」的数据能学起来

先看一个直觉问题：无监督学习里样本没有标注，看上去无从「学习」。但只要加入一条从业务里提炼的假设，学习目标就有了，过程就能跑起来。

- **聚类**：一堆没标注的点本来没法聚。但若假设「同类样本离本类中心比离别类中心更近」（K-Means 的假设），聚类就能展开；换一条假设「挨得近、聚得密的算同类，离得远、散的算不同类」（基于密度方法的假设），也能聚——两条不同假设可能给出相近结果，但出发点不同。
- **半监督**：假设「无标注样本与离它最近的有标注样本同类」（标签传播的假设），就能从有标注样本出发逐步把标签传播出去。

这些**对样本的先验假设，就叫归纳偏置**。要点：它本是数学 / 逻辑里的正式概念，但在数据科学中和具体函数模型解耦了，表现为一些**简单的逻辑表述**；这些逻辑不能凭空想，必须**基于真实业务场景提炼**。

### 监督学习也躲不开归纳偏置

有人以为：监督学习有标注做指引，分类 / 回归任务很明确，是不是就不用考虑假设了？答案是否定的，因为模型上线后总会遇到训练时没见过的情况：

1. **同样的特征，可能对应不同标注。** 当特征组合有限、而标注又多样时尤为常见。比如把同一款产品推给「同样画像」的两类人，必然有人成交有人不成交——标注并非一一对应。如何在这种不确定下让整体收益高、成本低，就取决于先验假设是否贴近现实。
2. **上线后会冒出训练时没出现过的特征组合。** 比如二维空间里训练时只有「方块类」和「圆形类」，上线却来了一个落在两类中间地带的新样本，该归哪类？没有现成标注能告诉模型答案，**此时决定分类结果的最重要因素就是归纳偏置**。

举例几种不同假设给出的不同边界：

- 有的模型假设「分界线到两类最近样本的距离应当相等」，于是画出居中的硬边界（一类最大间隔的思路）。
- 有的假设「最大间隙不必是直线，可以是柔性曲线」（高维映射的思路）。
- 还有人反驳：某类样本数量明显更多，分界线应当更靠近样本多的那一类才合理（带样本数量加权的思路）。

在有限的样本与特征空间里，往往**没人能拿出有说服力的证据**证明哪条假设一定对——这正是建模者常陷入犹豫的地方。

### 拿不准时：奥卡姆剃刀

当数据无法判定该信哪条假设时，可借**奥卡姆剃刀原理**——「如无必要，勿增实体」，即简单有效原理。它更像哲学式的指导原则，讲的是在充满未知的世界里，如何在「信息复用效率」和「不失优雅」之间找平衡。

应用到模型选择上非常直接：**如果决定不了选哪种假设，就选最简单的那种。** 比如上面区分方块与圆形、又没有更多样本和先验时，直接用「相等间隔的直线」做边界即可——既然数据给不出强证据，就让它简单些。

> 一条警戒线：用剃刀之前要先确认「实体」是否已经考虑齐全。奥卡姆剃刀不是「偷懒」的借口，而是在信息确实不足时的取舍原则。

### 把账算到「泛化能力」上

为什么归纳偏置这么重要？因为它直接关系到模型的**泛化能力**——即模型对「分析结论之外 / 训练时没出现过的样本」的预测能力，通常用**泛化误差**（模型的平均风险 / 期望损失）衡量。泛化误差与样本分布、模型输出都相关；一种方法若比另一种方法学到的模型泛化误差更小，它就更有效。

结论是清晰的：**模型泛化能力的强弱，与归纳偏置是否契合业务实际高度相关。** 反过来，如果样本数量足够多、特征分布足够丰富到能代表全体，样本分布就和总体一致，这时就不必在假设上花太多心思——可惜现实里这种理想情况不多。

### 几条常见归纳偏置（备查，并非全部）

- **最近邻**：特征空间里一小块区域内多数样本同类；给一个未知样本，猜它和最近的多数「邻居」同类。用于最近邻法的监督 / 无监督 / 半监督。
- **最大条件独立性**：最佳分类应是在样本特征条件下概率最大的结果。离散特征显著、可转成贝叶斯结构时适用（朴素贝叶斯的偏置）。
- **最大边界**：两类之间画线时，间隔最宽的分界最佳（一类最大间隔模型的偏置，假设类别由宽界线区分）。
- **加权最大边界**：在最大边界基础上，把各类样本数量作为边界宽度的加权因子。样本数量不均衡时可尝试。

> 提醒：归纳偏置来源于业务，它的多样性应被尊重——以上只是常见参考，不要当成清单去硬套。

## 四、用代理模型解释黑盒（高维场景挖掘与归因）

### 一句话

人最多同时想清楚两三个维度的交叉，再多就抓瞎。于是先让一个**预测力强的黑盒模型**把高维交互信息「学」进去，再用一个**人能看懂的简单模型（决策树 / 线性）去模仿这个黑盒**，从模仿过程里读出「在哪些维度组合（场景）下问题最严重」，并把它量化。预测解决「是多少（what）」，解释解决「为什么（why）」。这是把前面"函数模型 + 归纳偏置"落到高维归因场景的一种实战范式。

### 为什么不能只靠人工拆解

传统「经验化分析」是：分析师凭经验挑几个维度（性别、年龄、城市……）逐个、交叉地下钻看指标。它快、成本低，但有三个硬伤：

- 维度一多，交叉组合爆炸，人过滤无效组合的成本极高，三维以上基本超出直觉；
- 很难剥离其他因素的干扰，容易把「表面相关」当成原因；
- 多停留在定性、趋势层面，给不出「这个场景到底拉低了多少」的量化结论。

「体系化分析」用建模代替人工穷举：自动化、可复用、能快速加维度，还能挖出超越个人经验的高维交互。

### 黑盒 vs 白盒：先认清这对取舍

- **黑盒模型**（神经网络、梯度提升、复杂集成等）预测准，但内部机制不透明，说不清各特征的贡献和交互。
- **白盒模型**（线性回归、决策树）解释性强、能看清每个特征的影响方向和大小，但预测力相对弱。

关键认识：**可解释性不必是模型自带的属性**。即使用了黑盒，也可以借助专门的技术，在外面补一层解释——代价是一些额外的复杂度和算力。本框架正是这么做的。

### 核心工具：全局代理模型

「代理模型」是用一个简单模型去**复刻黑盒的决策行为**。注意它训练的目标很特别：

- 输入：同一批（或同分布的）特征 X；
- 标签：**不是真实的业务结果 y，而是黑盒对 y 的预测值 ŷ**。

也就是说，让简单模型学会「输入 X、输出和黑盒一样的 ŷ」。如果它学得像，就说明它在一定程度上「翻译」出了黑盒内部那套从 X 到 ŷ 的处理逻辑，于是黑盒就被解释了一部分。用一个类似回归 R² 的指标衡量「代理复刻得有多像」：越接近 1，说明它解释了黑盒预测里越大比例的变化；太低则说明这层解释不可靠。代理模型给的是**全局总结**，不可能完美还原黑盒的全部细节，但足以看清主干。

### 场景挖掘框架：四步走

1. **建黑盒做预测**：用预测力强的模型拟合目标指标，把高维交互吃进去。
2. **用决策树当代理**：在「特征 X → 黑盒预测值 ŷ」上训练一棵决策树。树天然按特征阈值一层层切分样本，每条「根到叶」的路径就是一个**维度组合的场景**（如「机型=低端 且 渠道=站外 且 时段=深夜」）。
3. **读出关键场景**：哪些叶子对应的指标表现特别差 / 特别好，就是值得重点看的场景。
4. （可选）**做量化归因**：把决策树切出的场景当成新的「交互特征」，和几个核心主因变量一起，对真实指标 y 建一个线性模型。这样就得到「各主因 + 各重点场景分别把 y 拉动了多少」的可量化结论，可直接服务决策。

> 中性示例：某内容平台发现「人均时长」下滑。先用梯度提升拟合人均时长，再用决策树模仿其预测，树切出一个明显偏低的场景——「新注册 且 仅用过站外入口 且 低端机型」；把这个场景做成交互特征喂进线性模型，估出它单独贡献了大约一半的下滑量，于是归因和后续策略都有了着落。

## 常见坑 / 边界

**关于三层模型观：**
- **三层混为一谈**：开会说「模型」时先问是哪一层——是在讨论业务逻辑、数据组织，还是函数关系？层次不对齐，沟通必然空转。
- **业务模型失真过度**：为了简化把和目标强相关的因素也砍掉，模型就失去指导价值。
- **只盯函数模型**：很多人一提模型就只想到算法 / 函数层，忽略前两层；而函数模型的好坏，往往取决于上游业务模型是否抓对了主要矛盾、数据模型是否把关系组织对了。
- **混淆参数与超参数**：参数是训练出来的，超参数是人工设定来控制训练过程的，两者的调法和角色完全不同。

**关于三种数据刻画：**
- **空间刻画忽略量纲与相关性**：不归一化、不处理特征相关，算出的距离没有业务意义。
- **概率刻画乱用乘法**：默认 P(A,B)=P(A)·P(B)，但只有特征独立时才成立，否则会高估或低估。
- **最优化只盯目标、忘了约束**：现实里几乎都有定义域和业务约束，漏掉约束求出的「最优解」可能根本不可行。
- **把视角当教条**：三种刻画是理解工具，不是非此即彼；方法好不好最终要靠实际业务检验。

**关于归纳偏置：**
- **在调参阶段才想假设**：归纳偏置该在**选 / 设计模型之前**考虑；有些模型本就是为某个偏置设计的，偏置和业务差太远，模型从一开始就失去意义。
- **凭空编假设**：归纳偏置必须从真实业务提炼，拍脑袋造的假设会把模型带偏。
- **把剃刀当懒惰**：先确认实体考虑齐全，再用「选最简单的」；否则是漏算，不是简化。
- **迷信单一假设**：同一份数据可用多条假设解释，不同假设给出的边界不同，要回到业务和泛化表现来判断。

**关于代理模型解释黑盒：**
- **代理像不像要先验收**：R² 类指标太低时，代理读出的「场景」可能是它自己的臆断，而非黑盒的真实逻辑，不能直接拿去汇报。
- **决策树切出的是相关结构，不是因果**：它告诉你「哪些组合下指标差」，但「为什么差」「改了会不会好」仍需业务判断或实验验证。
- **别把树喂过深**：树越深、叶子越碎，场景越不稳定、越容易过拟合，解释价值反而下降；适度剪枝，保留可解读的主干场景。
- **它是放大人、不是取代人**：框架负责从高维里筛出值得看的少数场景，最终的业务含义和行动，还得分析师结合场景理解来定。
