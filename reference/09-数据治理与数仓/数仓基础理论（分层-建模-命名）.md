## 正文

> ⚠️ 以下为改写后的自有表述，抽取概念与方法重新组织，不含任何出处、平台、作者、案例编号。表名/字段均为中性示例。

### 一句话定位
对数据分析师来说，理解数仓不是为了去写数仓任务，而是为了**听懂取数时数据开发在说什么**——知道一张表属于哪一层、是事实表还是维表、数据是增量还是全量、表名字段名的缩写代表什么。掌握这套语言，需求沟通、口径对齐、排查指标来源都会顺畅很多。

数据仓库的经典定义可以记成四个特征：**面向主题、集成、相对稳定、能反映历史变化的数据集合，用于支撑管理决策**。这句话很抽象，落到工作上可以用四个动作去理解：**采、建、管、用**。

- **采**：把各种数据源（MySQL、Kafka 等）的数据接进数仓统一处理。
- **建**：建设数据资产/模型，也就是开发明细表、宽表。
- **管**：通过指标体系、用户标签、数据门户、统一接口、ID 统一等手段管理数据，让业务能快速找到并用上数据。
- **用**：把加工好的数据交给下游创造价值——看板大屏、A/B 实验、专题分析报告等，支撑运营、风控等业务。

### 一、数据从产生到应用的完整流向

理解数仓前，先把"数据从哪来、到哪去"这条链路捋清楚。可以分成五步：

1. **行为产生与采集**：用户在 PC / 手机端的操作被记录下来，通过日志采集、操作记录入库。这一步通常由后端开发完成（客户端、服务端两侧都算）。
2. **数据接入数仓**：数仓把不同数据源（常见如 MySQL、Kafka）的数据接进来统一处理。
3. **清洗加工建宽表**：数仓对数据做清洗、转换（如把英文/数字枚举值翻译成中文）、**维度退化**（按看板和分析需要的角度，把维度数据补充进数据模型）、关联、指标开发，最终产出指标宽表、标签宽表。
4. **数据应用**：数据分析/算法团队基于这些宽表做实验、各类分析、可视化、模型开发。
5. **业务使用**：运营、风控等业务方拿分析好的数据去做投放、拉新、促活、风险拦截等动作。

数据分析师主要工作在第 4 步，但要明白自己手里的宽表是前三步层层加工出来的产物。

### 二、为什么不直接用数据库做分析：OLTP vs OLAP

常见疑问：为什么不能直接用 MySQL、Oracle 这类数据库做数据处理？本质区别在于两类系统定位不同：

- **数据库（OLTP，联机事务处理）**：面向日常事务，做记录的增删查改，以业务流程为主、更面向应用；不擅长存放与分析大量历史数据。
- **数据仓库（OLAP，联机分析处理）**：能大规模存放历史数据，更面向分析主题，适合做聚合、统计、多维分析。

简单说：**OLTP 管"当下的业务运转"，OLAP 管"历史的分析洞察"**。

---

### 三、两个容易混的概念：数据域 vs 主题域

数仓里有"面向应用"和"面向主题"之分，对应**数据域**和**主题域**两个概念。

- **数据域**：更贴近操作流程，记录每个业务流程中的明细数据，对应业务库里实际存在的数据。
- **主题域**：数据最终被用到的业务场景/分类。

可以用"做菜"来类比：
- **数据域**像做菜的**原材料**（萝卜、青菜、肉），对应业务库里的基础数据。
- **主题域**像**菜谱的分类**：大域好比菜系（川菜、粤菜、鲁菜），二级域好比具体菜品（回锅肉、烧腊）。

关键认知：**主题域一定是跨数据域的**。比如做回锅肉要用辣椒，做小炒肉也要用辣椒（共用同一原材料/数据域），但它们是两道不同的菜（不同主题域）。

### 四、数仓为什么要分层

分层不是为了复杂，而是为了四个收益：

1. **数据结构清晰**：每一层职责明确，用的时候好定位、好理解。
2. **血缘可追踪**：清楚每张表/任务的上下游，方便排查问题、找报错、找指标来源，也利于后期维护。
3. **减少重复开发**：把中间层做扎实，下游可以复用，减少重复造表、节省资源，同时保障口径和数据统一。
4. **把复杂问题简单化**：把一个复杂任务拆成多步，每层只处理单一步骤；出问题时只需从故障起点那一层开始修。

### 五、数仓各层都在做什么（ODS / DIM / DWD / DWM / DWS / ADS）

从最贴近数据源到最贴近应用，常见分层如下：

| 层 | 全称含义 | 主要职责 |
|---|---|---|
| **ODS（接入层）** | Operational Data Store | 最接近数据源的一层，从 API、业务库等把数据**原样同步**进数仓，中间几乎不做处理 |
| **DIM（维度层）** | Dimension | 集中管理维度，方便业务/下游查找和复用维度 |
| **DWD（明细层）** | Data Warehouse Detail | 对 ODS 数据做关联、清洗、**维度退化**（把维表中的维度放进明细表）、转换、主题域建设 |
| **DWM（轻度汇总层）** | Data Warehouse Middle | 介于 DWD 与 DWS 之间的过渡层，对 DWD 做整体关联，形成一张大的明细宽表 |
| **DWS（汇总层）** | Data Warehouse Summary/Service | 按主题域、颗粒度（如买家/卖家）、周期粒度、维度做聚合，形成指标较多的宽表，供业务查询和应用；**指标口径的统一与沉淀重点放在这一层** |
| **ADS（应用层）** | Application Data Service | 按应用域、颗粒度划分，把对应标签补充进来，最终形成用户画像、专项应用等成品 |

记忆主线：**ODS 原样进 → DWD 洗成明细 → DWM/DWS 汇总成宽表 → ADS 面向具体应用**。分析师取数时大多在 DWS / ADS 层取宽表，必要时回溯到 DWD 查明细。

> 提醒：DWM 这一层并非所有团队都设；DWS 的全称在不同资料里写作 Summary 或 Service，含义一致（汇总/对外服务）。

### 六、维度建模与五要素

**维度建模**：用**事实表 + 维度表**来构建数仓模型的方法，按维表与事实表之间的连接方式来开发数据表。建模时重点考量五个要素：

1. **数据域 / 主题域**：数据域按当前业务场景或业务 SOP 拆分来建设；主题域按业务的使用场景来定。
2. **事实表设计**：围绕业务过程设计，用描述业务过程的**度量**来表达这个过程，表里包含引用的维度和与业务过程相关的度量。
3. **维度**：对当前分析场景的角度做描述和补充（如类目、会员、区域等观察角度）。
4. **颗粒度**：在数据域下对场景做更细的拆分（如把"用户"拆成买家/卖家），且**必须拆到不可再拆的最小状态**（如用户拆到买家，买家不再细拆）。
5. **度量值**：场景下的数值型记录（如金额、数量、佣金等）。

**事实表的常见组成**（以一张"每日支付订单明细表"为例理解）：
- **业务过程**：订单支付。
- **事实表类型**：单事务事实表（一行对应一次事务）。
- **粒度**：子订单 ID（最细到每个子订单一行）。
- **度量**：订单支付金额、佣金、购买数量、各类补贴/费用等数值字段。
- **冗余维度属性**：把常用的维度直接冗余进明细表，如类目属性（一级/二级/叶子类目）、会员属性、区域属性（大区、县级组织）等。这正是上文"维度退化"的体现——为了查询方便，把维表字段拉平进明细表。

### 七、增量 vs 全量，以及数据分区

**数据分区**：分区是"某一天记录下来的数据样貌"，通常用日期做分区，方便快速定位数据。例如某天同步了 1000 条数据，那这一天的分区就有 1000 条；一张表会有很多天的分区。

在分区基础上，数据同步分两种方式：

- **全量数据**：每天把整个源库的**全部数据**都存进当天分区，对源数据全部覆盖。
  - 举例：前天分区存 1000 条订单，昨天分区存 1100 条——每天都是当时库里的全量快照。
- **增量数据**：每天只存**变更和新增**的数据进当天分区，对源数据做分区式覆盖。
  - 举例：前天分区存 1000 条（当天变更/新增量），昨天分区存 900 条——每天只是当天的变化量。

实务判断：维度小、变化频繁、需要历史快照的表常用全量；数据量大、只关心新增/变更的明细表常用增量。

### 八、命名规范速查（理解缩写就能读懂表名）

数仓表名、字段名通常遵循固定模板，掌握后看一眼表名就知道它是哪层、什么主题、什么粒度、什么周期。以下为通用化的命名约定示例（各团队会有差异）：

**表命名**（前缀即层级）：
- **ODS 层**：`ods_{业务库名}_{业务表名}_df/di`（`df` = 日全量，`di` = 日增量）
- **DWD 层**：`dwd_{数据域}_{二级数据域}_{业务过程}_df/di`（业务过程不明确时写 `detail`）
- **DIM 层**：`dim_{业务域}_{维度定义}`（无业务相关可省业务域；如日期维度写 `date`）
- **DWM 层**：明细宽表，命名思路同 DWD（`dwd/dwm_{数据域}_{二级数据域}_{业务过程}_df/di`）
- **DWS 层**：`dws_{数据域}_{二级数据域}_{颗粒度}_{业务过程}_{周期粒度}`（颗粒度如买家/卖家；无业务过程写 `target`；周期如近 30 天 `30d`、一个月 `1m`、90 天 `90d`）
- **ADS 层**：`ads_{一级主题域}_{二级主题域}_{颗粒度}_{业务过程}_{调度周期}`（一级主题域如风控 `risk`、营销 `mkt`；调度周期如每天一次 `1d`）

**字段命名**：
- 布尔/是否类：`is_{内容}`（如 `is_new_user`）
- 枚举类型：`{内容}_type`
- 时间类：日期用 `xxx_date`，时间戳用 `xxx_time`
- 周期指标：`{内容}_{时间描述}_date`（时间描述如最近一次 `last1`、最近两次 `last2`、历史 `his`、最近第二次 `last2nd`、最近 180 天 `180d`）
- 百分比/比率：`{内容}_rate`
- 数值-整型（计数）：`{内容}_cnt_{周期}`（周期视情况加）
- 数值-小数/金额：`{内容}_amt_{周期}`（周期视情况加）

读表名小技巧：先看前缀定层级（ods/dwd/dim/dws/ads），再看中间段定数据域和粒度，最后看后缀定周期与全量/增量（`df`/`di`）。

### 常见坑与边界
- **分层缩写与命名是约定俗成，不是标准**：ODS/DWD/DWS/ADS 是业界通行骨架，但各公司的层数、缩写（如有无 DWM）、命名模板都会有差异，沟通时先问清对方团队的规范。
- **维度退化要服务于查询，不是无脑冗余**：把维表字段拉进明细表是为了方便分析查询，但冗余越多维护成本越高，取舍看下游使用频率。
- **口径统一应落在 DWS 层**：指标口径如果各应用层各算各的，极易出现同名不同义；理想做法是在汇总层把口径定义好、沉淀下来，下游直接复用。
- **全量/增量影响历史回溯能力**：增量表只存变化量，想还原"某天的全量状态"需要做累积/拉链处理；分析历史快照前先确认手里的表是全量还是增量。
- **分析师不必会建数仓，但要会读数仓**：本篇是理论入门，目的是让分析师能与数据开发对齐口径、看懂血缘和表结构，而非取代建模工作。
