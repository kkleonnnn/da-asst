### 一句话定位
在分工明确的大厂，分析师拿到的往往是已经清洗好的宽表，建看板只需写几条 SQL、配一下；但在很多中小公司和创业初期，埋点、清洗、建模常会落到分析师头上。与其把这看成"不属于我的活"，不如把它当成一项能放大分析能力的技能——**懂一点数据治理，分析师就能在源头把口径定清楚，用埋点和数仓直接解决应用层很费力才能解决的问题**。

举个直观的例子：一个新上线的电商 App 想分析"用户点击下单前的上一个页面来自哪里"。如果只在应用层做，需要把用户的点击事件按时间排序再清洗计算，费时费力；但如果在埋点阶段就让程序在点击日志里加一个 `refer` 字段记录上一页 URL，那么无论是统计分析还是后续建表都能一步到位。这就是参与治理的价值——**问题在更上游解决，成本更低、口径更稳**。

> 一个常被问到的顾虑：参与治理会不会牺牲分析的深度？精力确实有限，专精分析、只挑数据建设好的团队也完全合理。但反过来，会一点治理能让择业范围更宽、在关键节点能更主动地借助数据开发的力量。两条路没有对错，本篇是给"想了解或不得不上手"的分析师做入门铺垫。

### 一、整条数据链路里，分析师该参与哪一段

参照一种通行的数据体系划分，整条链路自下而上分四层：

1. **数据采集层**：日志收集、业务库、数据同步。
2. **数据计算层**：离线计算、数据仓库、实时数据。
3. **数据服务层**：分析师平时能直接取数的地方——各类数据源（如 MySQL、Hive 数据源）、数仓的预聚合结果（cube）等。分析师的大部分工作就是拿这层的数据去做应用。
4. **数据应用层**：面向线上产品、对内系统等的成品数据。

纯技术向、涉及线上开发的内容通常不需要分析师参与。**分析师真正有发挥空间的两个环节是：日志埋点 和 数仓设计**，参与的深度随实际需求灵活变化。下面分别展开。

> 补充：业界对"数据管理"主题的经典分类（如数据治理、数据架构、建模、存储、安全、集成、质量、元数据等）很细，不同公司架构不一，没必要纠结术语，抓住"分析师能影响埋点和建模"这两个抓手即可。

### 二、日志埋点：分析师为什么要参与，以及怎么设计

#### 2.1 为什么不能把埋点全交给产品或开发

线上产出的原始数据通常是 JSON 或分隔符分割的字符串，里面定义了每个字段的 key-value，要经过清洗才能变成常用的数据表。分析师在埋点环节的核心职责，是**配合产品/运营，把每一条日志的上报机制、以及对应 key-value 的含义定义清楚**。

常见的反问是："这不该产品自己搞吗？"——基础能力强的产品确实能承担埋点，他最清楚功能上线后想看什么。但现实里常出现两种断层：

- 产品懂交互和功能设计，却不太懂数据，他要的往往是"用户活跃""漏斗转化"这种抽象概念，落不到具体字段。
- 于是产品去找开发，开发懂底层架构却不懂业务，如果没人把上报机制和字段含义定清楚，就容易埋错。

**一个口径错配导致误判的示意案例**：产品上了一个促销活动页，想看页面 PV 和带来的 GMV。开发图省事，埋了一个服务端日志——只要用户发了访问请求就记一条。结果统计出来 PV 巨大、GMV 却少得可怜。产品疯狂优化交互，GMV 依旧不动。多方排查才发现：前端页面加载有问题，很多用户请求了链接但页面素材没加载出来，而 PV 统计的是**服务端**日志，所以 PV 虚高。真实转化也许并不差，是**数据口径的差异**导致了整个判断失真。

这个案例说明：当产品或开发自身数据能力不够时，把埋点全权外包给他们，就容易出现口径不明确的问题。反过来，**埋点本就该为业务服务，最好的做法是从需求和数据指标反推该埋哪些点**。这正是分析师的参与空间，其意义有三：

1. 明确埋点对应的数据指标口径，避免业务分析偏差。
2. 帮助分析师了解底层架构，拓宽分析思路。
3. 分析师可以主动增加便于自己分析的埋点，提升效率。

#### 2.2 埋点的两大类方法

| 类型 | 做法 | 优点 | 缺点 / 适用 |
|---|---|---|---|
| **全埋点** | 部署 SDK 把 App 的全部行为都记下来，分析人员再自定义关键事件、直接圈选分析（常见于接入第三方数据服务供应商的系统） | 想怎么定义都行，无需重新开发 | 数据量大、占空间、存不久；只适合轻量级项目分析 |
| **代码埋点** | 让开发把关键事件信息写进代码，预先定义好"什么场景下返回哪些字段"。又细分为前端埋点与服务端埋点 | 精确、可控，是最常用的方式 | 需要开发投入，依赖事先把字段和场景定清楚 |

代码埋点的两个子类，差别关键在**准确性**：

- **前端埋点**：在 App 客户端或网页里，触发关键素材时上报，如页面加载、图片加载、按钮点击。**会受页面改版、网络等影响，存在误差**（上面 PV 虚高的案例本质就与前后端口径差异有关）。
- **服务端埋点**：成功请求某个服务器接口时上报，如下单、播放视频。**通常最准确**——请求成功了就是成功了，不受前端改版影响。

实务取舍：要"是否真正发生了业务动作"（下单、支付、播放）就走服务端；要"用户看到/点到了什么界面元素"（曝光、点击）才用前端，并意识到前端数有天然误差。

#### 2.3 埋点设计的五个步骤

设计埋点建议遵循固定流程，让产品、分析、开发三方都听得懂：

1. **明确业务需求**：先和产品/运营对齐，看产品文档——新功能/改版做了什么改动，是否需要前端或服务端埋点。
2. **明确数据指标**：功能上线后要看哪些核心指标，分别要在前端、服务端埋哪些内容，确保上线能统计到。
3. **梳理埋点字段**：按字段大类系统地列出要埋的字段（见 2.4 的清单）。
4. **明确上报机制**：和开发对齐——在什么情况下上报、哪些字段能上报、哪些字段记录不了该怎么处理。这些定清楚才能进入开发。
5. **提交数据需求文档**：输出成型的埋点文档（见 2.5 的结构）。

核心逻辑是：**从产品对 UI 的理解 → 过渡到数据指标的设计 → 再到具体开发**，三方认知拉齐，沟通才高效准确。

#### 2.4 埋点字段设计清单（按大类梳理）

设计字段时，按下面几个大类逐一过一遍，不容易漏：

| 维度 | 包含信息 | 备注 |
|---|---|---|
| **日志基础信息** | 日志唯一标识、日志 id、事件 id、事件类型等 | 常用作日志的分区字段 |
| **页面信息** | 名称、title、模块、链接等 | 一般前端需要的较多 |
| **用户基础信息** | 用户 id、设备信息（设备号、型号）、操作系统（语言、版本）、网络信息（如 ip）、应用信息（版本、包体）等 | 有些敏感信息不一定能拿到；用户明文账号等要注意加密 |
| **时间信息** | 日志上报时间、上传时间、更新时间、创建时间 | 一次性事件记上报时间即可；可累积更新状态的对象（如订单）要记录各状态的时间 |
| **业务关键信息** | 非公共参数，按业务定义：关注增长就记点击来源/渠道；关注停留消费就记时长、下单金额；有"用户-对象"交互（用户-商品、用户-视频）就记商品 id、视频 id 等 | 由业务差异决定，最灵活的一块 |
| **拓展字段** | 留一个空的 `desc` 或 `info` 字段 | 未来有新增需求时，以 JSON 字符串形式在此扩展，避免频繁改表 |

> 用这套标准写埋点文档的好处是：能拉齐各方对埋点的认知，沟通更高效、更准确。

#### 2.5 成型埋点文档的结构

最终的埋点文档建议把"日志基础信息 + 页面信息 + 具体字段 + UI 图"对齐到一起，便于三方核对。可以理解成一张表，每个关键事件一行，列大致包括：

- **事件 / 事件类型**：如"首页浏览 / `page_view`""曝光"等。
- **页面信息**：名称、模块（如"首页"）。
- **记录字段 / 记录值**：公共字段（用户 id、设备号、时间、页面 id 等）+ 业务字段（如该页面属于某个游戏或商品，就记游戏 id / 商品 id）。
- **UI 图**：贴上对应界面截图，让产品和开发能对着图确认埋点位置。

### 三、数仓设计：分析师在哪一层该发声

#### 3.1 为什么从零搭建数仓时分析师要参与

数仓一般和数据存储、数据安全绑定，多数工作落在数据开发身上——**前提是数据体系已有沉淀**。但如果是从 0 到 1 搭数仓，分析师的参与空间很大。

原因在分工互补：数据开发擅长把数仓做得高效、可扩展、可维护（比如把上亿条"用户-视频"事实表的清洗任务耗时砍半），但到了服务层、应用层，**怎么贴合业务**就需要分析师。例如要定义"近 30 天用户活跃天数""近 90 天用户观看时长"这类指标，数据开发未必知道怎么设计才贴业务，这时就该分析师介入。

#### 3.2 分层概念与分析师的介入点

数仓自下而上的常见分层（此处仅作介入点说明，分层与命名的系统讲解见同板块《数据分析师视角的数仓基础理论-分层建模与命名规范》）：

- **ODS（数据准备层）**：接入数据源的原始日志。
- **DWD（数据明细层）**：把 ODS 的原始 JSON 等字符串解析成数据库表，保持与 ODS 同样的颗粒度，并做空值填补等清洗。
- **DWM（数据中间层）**：在 DWD 基础上做轻微聚合，算出统计指标。例如短视频产品 DWD 记的是"用户-创作者-作品-时长"，同一用户多次看同一视频会有多条记录；DWM 可按需聚合为"用户-创作者-时长"，每对用户-创作者只留一条。
- **DWS（数据服务层）**：在 DWM 基础上整合的主题表，如"用户主题表：用户-总时长-创作者人数""创作者主题表：创作者-用户数-总时长"，维度已具备业务含义。
- **ADS（数据应用层）**：提供给产品或分析的表，如 BI 指标表、为快速分析预聚合好的表（如"近 90 天观看视频数"），自定义程度更高。

一个直观的理解：**数仓从下往上就是一个不断 `group by` 的过程**——从多条明细 group 成一条，从 N 个维度 group 成一两个维度。**"如何选维度、group by 出哪些指标"，正是分析师发挥作用的地方。**

具体介入点：

- **ODS、DWD**：通常无需分析师参与，数据开发保证准确性和稳定性即可。
- **DWM 起，分析师可以介入选维度**。举例：DWM 待聚合的维度有 20 来个（用户 id、创作者 id、视频信息、机型、IP……），分析师可结合分析经验挑该保留哪些——如果 IP、机型在日常分析里不是主维度，DWM 就不必保留；但如果常统计"活跃设备数"，设备 id 就要留下来。

#### 3.3 维度建模的四个步骤（Kimball）

设计事实表时可按维度建模四步走（以直播产品为例）：

1. **选择业务过程**：确定业务场景。比如把核心路径定义为"观看直播-付费充值-礼物打赏"，那最初的事实表要先定是单一场景（只看"观看直播"）还是多场景叠加（"观看直播+付费充值"）。
2. **声明粒度**：确定主键。如在"观看直播行为表"中选用户作为粒度。
3. **确认维度**：按常用分析维度挑字段。以用户为粒度时通常关注"看了哪些主播、在什么渠道看、看什么类型"，维度就包含主播 id、渠道来源、直播品类——核心考量是业务相关性。
4. **确认事实**：确定度量指标。如观看直播场景要关注时长、PV，就在聚合时把这两个指标算出来。

无限拓展下去维度可以拆很多，常见三种模式：

| 模式 | 特点 | 维护难度 |
|---|---|---|
| **星形模式** | 以事实表为中心，所有维表直连事实表 | 低 |
| **雪花模式** | 维度表可以再挂其他维度表 | 高 |
| **星座模式** | 基于多张事实表，共享维度信息 | 高 |

无论哪种，核心都是**在存储空间和业务便捷性之间找平衡**：维度表越多，分析越方便，但存储成本越高；维度越简单，数仓越高效，但稍复杂的分析就得从最底层表搭起，降低效率。这块需要分析师（出业务视角）和数据开发（出技术方案）长期共建，单靠一方都难做好。

### 四、治理与分析的"共同进化"

参与治理的更大意义，是给分析师提供一种**跳出纯分析框架解决问题**的思路：懂一点治理方法，就能在关键节点借助数据开发的力量。两个方向的正反馈：

- **分析发现 → 反推治理**：比如在用户路径分析中发现"用户浏览商品详情页 3 次以上、购买率提升约 10%"（示意数字），就可以设计对应埋点——每次曝光商品时记录"当天已曝光该商品的次数"，产品据此做干预策略。又比如运营总要看不同机构（如 MCN）维度的数据，而数仓原本只有"用户-创作者-视频"事实表，那就可以给数据开发提需求，增加字段或新建事实表/维度表，方便后续建表。
- **治理沉淀 → 反哺分析**：数据开发也能从中获得业务经验反哺。发现"商品曝光次数"是关键行为，下次打其他埋点时就可以建议产品带上；发现业务方常看 A 维度不看 B 维度，就可以设计更便捷的维度表给他们用。

整体上，治理是分析与数据开发**相辅相成、共同进化**的协作：分析师提供业务视角的建议，数据开发提供技术方案。

### 常见坑与边界
- **口径错配比数据缺失更隐蔽**：PV/GMV 案例的本质是"前端 vs 服务端"口径差异导致误判。埋点前先想清楚每个指标该用哪一侧的日志，准确性优先选服务端。
- **前端埋点天然有误差**：受改版、网络、素材加载影响，分析前端曝光/点击数时要留一份"它可能偏高/偏低"的心理预期。
- **埋点要从指标反推，不要从功能正推**：先问"上线后要看哪些指标"，再决定埋哪些点；让产品/开发自由发挥容易埋出口径不清的点。
- **敏感字段必须脱敏/加密**：用户明文账号等敏感信息要么拿不到，要么必须加密处理，不能直接进日志。
- **拓展字段是省心的伏笔**：预留 `desc`/`info` 字段用 JSON 扩展，能减少未来频繁改表。
- **分析师参与有边界**：ODS/DWD 的准确性稳定性归数据开发，分析师主要在 DWM 及以上贡献"选维度、定指标口径"的业务判断，不必也不该取代建模本身。
- **本篇为入门视角**：目的是让分析师能上手参与埋点设计、看懂数仓并在关键处发声，而非系统的数仓开发教程；分层与命名的细节见同板块的数仓基础理论篇。
