## 正文

> ⚠️ 以下为改写后的自有表述，抽取方法与链路逻辑重写，不含任何出处、平台、作者、案例编号。SQL 仅给出**结构示意**，表名/字段为中性示例。

### 一句话方法论
一个 A/B 实验平台的后端数据，本质是把**两路源数据**（平台侧的实验配置、客户端侧的用户行为日志）按数仓分层逐级加工：先各自落明细，再关联成"带实验标识的用户行为明细"，在此之上算指标，最后对指标做统计检验，输出"实验是否成功"的判定。整条链路可以拆成三层：**数据采集层 → 指标计算层 → 统计检验层**。

### 整条链路的结构
把链路想成一条从左到右的流水线：

1. **数据采集层（日志/明细层）**：把两路源数据收进数仓，整理成明细表。
2. **指标计算层**：基于明细按实验组聚合，算出各类指标（事件型、漏斗型、留存型）。
3. **统计检验层**：根据样本量选 Z 检验或 T 检验，算检验统计量与 p 值，判断显著性，最终判定实验成败。

每一层的产出是下一层的输入，分层后职责清晰、便于排查和复用。

### 一、数据采集层：两路源数据如何进数仓

实验数据有两个来源，需要分别处理后再关联。

**源 A：实验配置信息（来自实验平台）**

- **产生处**：实验平台本身。
- **内容**：实验名称、实验关键指标、实验创建人、实验指标对应的埋点信息等"元数据"。
- **作用**：决定一个实验"该看哪些指标、各指标对应哪些埋点"，是后续把日志翻译成指标的字典。
- **采集与存储**：平台后端常用文档型数据库（如类 Mongo 存储）记录配置，需要先把它抽到数仓，再拆成几张维度/明细表，例如：
  - 实验版本表（记录每个实验有哪些版本/分组）
  - 实验清单表（实验基础信息）
  - 实验指标表（每个实验关注的指标及其口径）
  - 实验项目表（实验所属项目）
- **加工要点**：把分散在不同配置表里的信息**关联整合成一张宽表**，便于后续与行为日志做关联。关联键通常是实验 id、版本 id、指标 id、项目 id 这几条主键串联（实验—版本、实验—指标、实验—项目逐层 join）。

结构示意（仅示形）：

```sql
-- 把实验清单 / 版本 / 指标 / 项目逐层关联成实验配置宽表
select ...
from   实验清单表 p1
left join 实验版本表 p2 on p1.lab_id      = p2.lab_id
left join 实验指标表 p3 on p1.indic_id    = p3.indic_id
left join 实验项目表 p4 on p1.proj_id     = p4.proj_id
```

**源 B：用户行为日志（来自客户端埋点）**

- **产生处**：客户端用户的行为埋点。
- **内容**：用户的埋点行为（位置、动作、时间）以及用户标识。
- **作用**：是指标计算和检验的"事实数据"。
- **采集与存储**：落成"前端用户行为日志明细表"。
- **加工要点**：埋点日志里通常用一个拼接字段携带实验信息（如把"版本/实验/分组"等编码在一个字符串里），需要**按分隔符拆解**还原出"用户命中了哪个实验、哪个版本"。同时按用户、按埋点统计次数，得到"某用户在某埋点上的行为次数"。

结构示意：

```sql
-- 从埋点日志解析实验标识，并按用户 + 埋点聚合次数
select user_id,
       stat_id,                                   -- 埋点 id
       count(1) as cnt,                            -- 行为次数
       split(logmap['version'], '_')[1] as lab_id,      -- 解析出实验 id
       split(logmap['version'], '_')[2] as lab_version  -- 解析出版本号
from   行为日志明细表
where  日期分区条件
  and  split(logmap['version'], '_')[4] is not null      -- 过滤无实验标识的日志
group by ...
```

**两路关联成"实验数据明细"**

把"实验配置宽表"里的**埋点信息**与"行为日志"里的**用户行为**按埋点关联，就得到一张核心中间表：**带实验/版本标识的用户行为明细**。这张表是整条链路的枢纽——它既知道"这条行为属于哪个实验的哪个版本"，又知道"这个行为发生了多少次"，供后面所有指标计算和检验使用，落在 ADS 层供检验直接取用。

> 落地提醒：埋点信息的具体存储形式（放配置库还是落数仓）往往需要数据开发与平台开发协商确定；拆字段的规则要和埋点上报的编码规则严格对齐，否则实验标识会解析错。

### 二、指标计算层：三类指标的算法

实验指标按统计逻辑分三类，前两类有成熟解法，第三类（留存型）思路给出但需自行展开。

**1）事件型指标（UV / PV 型）**

- **含义**：直接对某类事件计数。UV 型按去重用户数算，PV 型按行为次数（事件数）算。例如"点击某按钮的人数（UV）""点击某按钮的总次数（PV）"。
- **算法**：在用户行为明细上，按实验+版本分组，用 `case when` 区分指标类型，PV 型对次数求和、UV 型对用户去重计数。一张 SQL 里用多个 `when index_type = '...' then ...` 分支同时算出多个事件指标。

结构示意：

```sql
select lab_id, lab_version,
       sum(case when index_type = 'PV型' then cnt else 0 end)            as pv_metric,
       count(distinct case when index_type = 'UV型' then user_id end)    as uv_metric
from   实验用户行为明细
group by lab_id, lab_version
```

**2）漏斗型指标（UV/UV、PV/PV 型）**

- **含义**：一个比率，分子分母都是事件型指标。如"完成支付 UV ÷ 进入收银台 UV"是 UV/UV 型转化率；"点击次数 ÷ 曝光次数"是 PV/PV 型。
- **算法**：先分别算出分子事件和分母事件（复用事件型指标的算法），再相除。关键是**分子分母都要按同一实验+版本口径对齐**，避免错配。

结构示意：

```sql
select lab_id, lab_version,
       分子事件量 / nullif(分母事件量, 0) as funnel_rate
from ( ... 在明细上分别聚合出分子事件量、分母事件量 ... )
```

**3）留存型指标（次日留存 / 当日 UV 型）**

- **含义**：跨天的留存类指标，如次日留存率。
- **思路**：需要把"某日进组用户"与"次日仍活跃用户"跨天关联后求比例。这类指标涉及多天数据 join，复杂度高，属于在前两类基础上的扩展，落地时按业务定义自行展开。

### 三、统计检验层：选检验、算统计量、判显著、定成败

指标算完后，要判断"实验组与对照组的差异是真实的还是抽样波动"，这就是统计检验。

**1）按样本量选检验方法**

- **大样本（样本量 > 30）→ Z 检验**
- **小样本（样本量 < 30）→ T 检验**

这里的"样本量"含义因指标类型而不同（见第 4 步的判定逻辑）。

**2）两种检验的统计量公式**

- **Z 统计量**（两组均值差，除以合并标准误）：

  $$z = \frac{\bar{x}_1 - \bar{x}_2}{\sqrt{\dfrac{s_1^2}{n_1} + \dfrac{s_2^2}{n_2}}}$$

  其中 $\bar{x}_1, \bar{x}_2$ 为两组样本均值，$s_1, s_2$ 为标准差，$n_1, n_2$ 为样本容量。

- **T 统计量**（小样本，使用合并方差）：

  $$t = \frac{\bar{x}_1 - \bar{x}_2}{\sqrt{\dfrac{(n_1-1)s_1^2 + (n_2-1)s_2^2}{n_1+n_2-2}\left(\dfrac{1}{n_1}+\dfrac{1}{n_2}\right)}}$$

  符号含义同 Z 检验。

**3）落地计算的关键步骤**

- 先在指标明细上**按实验+指标聚合出每组的样本量、均值、方差**（均值/方差用聚合函数算，比例类指标的方差可由 $P(1-P)$ 直接得到）。
- 把对照组与实验组的统计量拼到一行，代入公式算出 Z 或 T 值。
- T 检验还需根据**自由度**查临界值表——工程上常把临界值表用一长串 `case when df = ... then 临界值` 内联进 SQL，避免外部查表。

**4）判显著与判实验成败**

- **显著性判定**：以 p 值阈值 0.05 为界。算出的 Z/T 统计量若超过对应临界值（即 p < 0.05），判为"有显著差异"，否则"无显著差异"。在 SQL 里用 `case when 统计量 >= 临界值 then 1 else 0 end` 标记是否显著。
- **实验是否成功的判定逻辑**（关键在于"样本量按什么单位算"）：
  - **关键指标是漏斗型**：样本量按**人**算（样本量通常 > 30）→ 走 **Z 检验**判定。
  - **关键指标是事件型 / 留存型**：样本量按**日期（天数）**算（天数通常 < 30）→ 走 **T 检验**判定。
  - 最终看**关键指标**的检验结果是否显著且方向为正，来判断这次实验是否成功。

### 常见坑与边界
- **拆解埋点字段的规则必须与上报编码严格对齐**：实验标识藏在拼接字符串里，分隔符或位置错一位，整条链路的实验归属就全错。
- **分子分母口径要对齐**：漏斗型指标的分子、分母必须在同一实验+版本粒度上统计，否则比率失真。
- **"样本量"的单位随指标类型变**：漏斗型按人、事件/留存型按天，直接决定该用 Z 还是 T，不能一刀切。
- **这是单场景单方法的最小版本**：只覆盖单一检验路径，留存型指标仅给思路；多场景、分位数/比率类指标、实时性要求高的链路都需要在此框架上扩展。
- **配置数据与日志数据要解耦再关联**：两路源数据各自先落明细、各自可独立排查，再在中间层关联，便于定位"是配置错了还是日志解析错了"。
- **显著 ≠ 业务有价值**：检验只回答"差异是否非偶然"，效应量是否值得推全量要结合业务体量另判。
