# 协作指南：怎么喂料、怎么脱敏消化、怎么让它正式生效

> 给不写代码的伙伴：这个仓库就是一堆文本文件，**贡献 = 喂资料 + 提 PR，不需要写代码。**
> 先读根目录 `README.md` 了解全局，再看这里的具体操作。

## 一、喂料（任何人、任何工具都能做）

把外部资料放进 `inbox/`：
- 可以建日期子文件夹，例如 `inbox/2026-06-22/`。
- 命名建议：`日期 主题 简述`（主题用 `_` 连接层级），例：`2026-06-22 分析方法_留存分析 cohort留存怎么看.md`
- **三条铁律**：①一条只讲一件事 ②不放真实业务数据 / 密钥 / 个人隐私 ③**原文只留本地**。

> 📌 **整个 `inbox/` 已被 `.gitignore` 挡住——原文和第一遍 `.md` 都不会进 git。** 想同步给协作者，**别传原文 / 第一遍稿**；只有最终整合进 `reference/` 的脱敏文本才随 git 同步。

## 二、两遍消化制（第一遍人人可做，最终整合是质量 + 防溯源关口）

**第一遍消化（任何人、任何模型，本地）**：把你准备的资料，**本地用你自己的 AI 先粗消化一遍** → 产出结构化 `.md`（提炼要点、注明建议归入哪个板块）→ 放进 `inbox/`。
> ⚠️ **第一遍 `.md` 只留本地、绝不提交**：`inbox/` 全量被 `.gitignore`，且第一遍稿尚未脱敏（可能仍带原文与来源痕迹）。它只是给最终整合省力的草稿。

**最终整合（写入 `reference/`）= 质量 + 防溯源关口**（建议 Opus 4.8+ 把关）：把 inbox 的资料 / 第一遍 `.md` 做下面这套硬功夫后并入 `reference/` 对应板块。**其他 AI、弱模型、人工都不要直接改 `reference/`**——只交第一遍 `.md` 到 `inbox/`。

**最终整合硬规则（防溯源，逐条遵守）：**
1. **彻底改写**：用自己的话重述事实与方法，打散原文的标题 / 句式 / 比喻 / 案例编号；**不得整段照搬**。改写后即使有人拿文本去搜索引擎反搜，也搜不到原帖。
2. **只留知识、不留来源**：删除作者、ID、平台 / 社群名、链接、二维码、截图、付费 / 入群引导等**一切来源指纹**。
3. **元信息只写** `适用范围 / 时效 / 可信度 / 方法论标签`（见 `templates/知识文件模板.md`），**不写出处**。
4. **交叉核实**：能验证的事实尽量核实；不确定的标「可信度低 / 存疑」，不要把营销话术当定论。
5. **跑 scrub + 记日志**：写入前跑 `bash scripts/scrub_check.sh` 自检；完成后在 `reference/_消化日志.md` 追加一行（批次 / 日期 / 进了哪些板块 / 模型），**日志同样不写来源身份**，批次用中性编号或日期。

> 为什么分两遍：第一遍**众包减负**（任何模型都能粗消化）；最终整合是**质量 + 防溯源关口**，保证 `reference/` 干净、可信、不可溯源。

## 三、生成 + 自检

```bash
bash scripts/build_pack.sh    # 由 reference/ 重新生成 通用知识包.md，并自动跑禁词扫描
bash scripts/scrub_check.sh   # 单独跑来源指纹 / 禁词扫描（命中即报错）
```

> **首次使用**：复制 `scripts/.scrub_wordlist.example` → `scripts/.scrub_wordlist.local`，把要拦截的来源相关词（平台名、作者 ID 等）逐行填进去。该文件已 `.gitignore`、**留本地不入库**——所以真实来源词只存在于你本地，公开仓库里任何文件都不会出现它们。

## 四、提 PR + 合并 main（正式生效的关口）

**资料消化进 reference/ 后，不会自动生效，必须走这一步：**
1. 把改动提一个 **Pull Request**（不要直接推 main）。
2. 审核后**合并到 main**——这才算"正式迭代"。
3. 另一方 `git pull` 即可拿到最新知识。

> 不写代码也能提 PR：用 **GitHub 网页**（编辑 `.md` → 选择"新建分支并发起 PR"）或 **GitHub Desktop**，全程点鼠标。

## 五、消费（任何 AI 都能做）
把 `reference/` 或 `通用知识包.md` 喂给任何 AI 当上下文，让它回答分析问题、设计指标、选分析方法、出复盘 / 分析报告。详见 `README.md` 的「给非 Claude AI 的使用说明」。

## 六、新增「行为规则」（应答风格 / 红线等，区别于"知识"）

「知识」和「行为规则」分两条路放，别混：

- **知识**（方法、框架、口径、案例）→ 进 `reference/`，走上面的两遍消化制。
- **行为规则**（怎么答、应答风格、安全红线等，如「应答原则」）→ **不进 `reference/`**（那不是知识），按这套来：
  1. 写进 **`SKILL.md`** 对应原则节 —— Claude skill 触发时自动生效。
  2. 若想让**非 Claude 模型**也遵守 → 另抽一份 **model-agnostic 的可粘贴片段**（独立 `.md`，参考 `决策应答原则-可粘贴提示词.md`），并在 `README.md` 的跨模型说明里指路。

> 为什么分开：行为规则要靠运行环境（harness）注入才生效；`通用知识包.md` 只由 `reference/` 生成、**只装知识不装行为**，要保持纯净。所以行为规则不进通用包，非 Claude 模型靠"粘贴片段"获取。

## 目录速查
| 文件夹 / 文件 | 放什么 |
|---|---|
| `inbox/` | 待消化的外部资料（**整个不入库，只留本地**） |
| `reference/` | 已消化、已改写、已脱敏的正式知识 |
| `通用知识包.md` | 由 reference 生成的整合文件（勿手改） |
| `templates/知识文件模板.md` | 每个知识文件的写法 |
| `scripts/.scrub_wordlist.local` | 你的真实禁词清单（不入库） |
